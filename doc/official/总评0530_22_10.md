# BuyPilot-AI 后端评委审查报告

审查时间：2026-05-30 22:10  
审查范围：后端为主，兼顾端到端契约与客户端消费风险。  
审查原则：只按可验证代码、配置、测试和本次运行结果评分；README/文档仅作为线索。

## 一、总评

这个后端已经像一个可参赛交付，不是空壳 mock：核心 `/chat/stream`、商品入库、chunk embedding、向量召回、rerank、证据绑定、SSE 商品卡片、购物车闭环都有代码事实支撑。

最大优点是 RAG 链路分层清楚，商品事实源和 LLM 生成边界分得比较硬。最大风险是“演示级证明”还不够稳：本机 live smoke 通过，但用的是 SQLite 持久化向量路径，不是 pgvector；完整 `demo_smoke` 运行 7 分多钟无输出，被审查时终止。

若我是评委，会相信它是真 RAG Agent，但会追问 pgvector 实跑证明、预算放宽策略、多模态到底是不是视觉相似检索。答辩时最容易被抓的是 README/CLAUDE 里的模型策略和代码配置不一致。

## 三、硬伤清单

### P0

#### 1. 完整 demo smoke 本次未完成

- 问题描述：`./.venv/bin/python -m src.scripts.demo_smoke --no-write` 运行约 7 分多钟无输出，审查时终止进程。
- 代码位置：`backend/src/scripts/demo_smoke.py:172`
- 为什么违反课题要求：评委验收需要可复现 Demo 路径；脚本卡住但不输出卡点，会直接影响现场排障。
- 建议修复方式：每个 turn 加 `asyncio.wait_for` timeout、逐场景 start/complete 日志、失败时输出已收 SSE events。
- 修复优先级：P0。

#### 2. live smoke 不强制 pgvector

- 问题描述：本机 `.env` 的 `DATABASE_URL` 是 SQLite，`smoke_live_rag` 仍通过；脚本没有断言 PostgreSQL/pgvector。
- 代码位置：`backend/src/scripts/smoke_live_rag.py:49`
- 为什么违反课题要求：README 推荐评委验收用 PostgreSQL + pgvector，但当前门禁不能证明实际使用了 pgvector。
- 建议修复方式：增加 `--require-postgres` / `--require-pgvector` 参数，打印 engine dialect，并在评委门禁中强制启用。
- 修复优先级：P0。

### P1

#### 3. 模型策略文档与代码不一致

- 问题描述：文档声称 Doubao 做意图主力、Qwen-Plus 做生成主力；代码当前 `TASK_MODEL_MAP` 中 `analyze_intent/generate_*` primary 都是 `qwen_turbo`。
- 代码位置：`backend/src/config/settings.py:39`
- 为什么违反课题要求：答辩时模型选型是高频追问点，不一致会削弱可信度。
- 建议修复方式：统一代码或文档；如果当前以 Qwen-Turbo 为主，应更新 README/CLAUDE/design-decisions。
- 修复优先级：P1。

#### 4. 预算放宽可能静默违反用户约束 ✅ 已修复

- 问题描述：严格过滤无结果时，检索会放开 `budget_max`，但卡片和文案不一定明确提示”预算已放宽”。
- 代码位置：`backend/src/services/retriever.py:273`
- 为什么违反课题要求：价格约束是购物推荐硬约束，静默超预算容易被判为不可靠。
- 建议修复方式：将 relax 步骤透传到 `risk_notes`、criteria card 或 text_delta，明确”原预算内无匹配，以下为超预算备选”。
- 修复优先级：P1。
- **修复状态**：已修复。新增 `_budget_was_relaxed()` 从 `trace_details.filters_applied.relaxation_steps` 提取预算放宽信号；放宽时每张 `ProductCardEvent.risk_notes` 注入”原预算内无匹配，此为超预算备选”；followup text 替换为”原预算内没有匹配商品，以下为放宽预算后的备选...”。

#### 5. 测试存在内部 mock 白名单

- 问题描述：测试守卫允许若干内部函数 mock，包括 `_chat_completion`、`_embedding_request`、`_rerank_request`。
- 代码位置：`backend/tests/test_architecture_layers.py:98`、`backend/tests/conftest.py:58`
- 为什么违反课题要求：单元测试能证明编排和规则，但不能充分证明 provider transport 和真实 schema 兼容。
- 建议修复方式：优先改到 `httpx.AsyncClient` 边界 mock；live gate 单独证明真实 provider。
- 修复优先级：P1。

### P2

#### 6. `auto_run` 字段未被后端消费 ✅ 已修复

- 问题描述：请求 schema 接收 `auto_run`，但 runtime 未搜索到实际行为分支。
- 代码位置：`backend/src/types/schemas.py:27`
- 为什么违反课题要求：接口字段没有语义会让前后端契约变得含糊。
- 建议修复方式：实现语义或删除字段，并同步 Android/contract 文档。
- 修复优先级：P2。
- **修复状态**：已修复。删除 `ChatStreamRequest.auto_run` 字段（无语义的字段不值得保留），同步清理 9 个测试文件中 16 处 `auto_run` 引用。contract JSON Schema 和 Android Kotlin 中原本就没有此字段，无需同步。

#### 7. 多模态不是视觉相似召回

- 问题描述：当前多模态能力是 VLM 属性抽取后拼入文本检索，不是 image embedding 或视觉相似检索。
- 代码位置：`backend/src/runtime/stages/multimodal.py:3`
- 为什么违反课题要求：如果答辩描述成“以图搜同款/视觉相似找货”，会被追问实现细节。
- 建议修复方式：答辩口径说清楚“VLM 提取可见属性驱动 RAG”；若冲高分，补 image-to-product 相似召回。
- 修复优先级：P2。

#### 8. 商品 SKU 未映射到 runtime payload ✅ 已修复

- 问题描述：`ProductPayload` 有 `sku_options`，raw 数据也有 `skus`，但 `_payload_from_raw` 未填入。
- 代码位置：`backend/src/repos/products.py:97`
- 为什么违反课题要求：商品卡片可消费性还可以更完整，规格信息对加购/对比有用。
- 建议修复方式：将 raw `skus` 规范化写入 `sku_options`，并加契约测试。
- 修复优先级：P2。
- **修复状态**：已修复。新增 `_normalize_sku_options()` 将 raw SKU 数据（`sku_id` + `properties` dict + `price`）规范化写入 `ProductPayload.sku_options`；添加 7 个契约测试覆盖正常路径和边界（空列表、None、缺失 sku_id、缺失 properties），包括 JSON Schema 校验验证 SKU 在 SSE 事件中合规传输。

## 四、接口与链路核查

核心链路：

```text
用户请求
→ backend/src/api/chat.py:/chat/stream
→ backend/src/runtime/pipeline.py:chat_stream
→ intent / criteria / retrieval stages
→ backend/src/services/retriever.py:retrieve_with_evidence
→ backend/src/services/llm_client.py:generate_recommendation / stream_recommendation
→ backend/src/runtime/handlers.py:_product_card_events
→ backend/src/types/sse_events.py:format_sse
→ Android SSE parser / ChatUiNode
```

核心接口：

| 接口 | 能力 | 请求/响应判断 |
|---|---|---|
| `POST /chat/stream` | 核心对话与推荐 SSE | 请求含 `message/session_id/history/image_url/criteria_patch/client_turn_id/client_trace_id`；响应为 `text/event-stream` |
| `POST /upload/image` | 图片上传 | multipart 图片，返回 `image_url/width/height/mime_type` |
| `POST /feedback` | 反馈记录 | 写入 session/deck/product 反馈 |
| `POST /chat/cancel` | 流式 turn 取消 | 支持进程内 token + DB cancel request |
| `GET /cart/{session_id}` | 查询购物车 | 返回 item、数量、总价 |
| `PATCH /cart/{session_id}/items/{product_id}` | 修改数量 | 结构化 CRUD |
| `DELETE /cart/{session_id}/items/{product_id}` | 删除商品 | 结构化 CRUD |

SSE 事件类型：

- `thinking`
- `clarification`
- `criteria_card`
- `text_delta`
- `product_card`
- `cart_action`
- `final_decision`
- `done`
- `error`

客户端可消费性判断：

- 支持流式：是，`StreamingResponse` + SSE。
- 返回商品卡片：是，`product_card`。
- 有 session/conversation_id：有 `session_id`，conversation DB 行内部持久化。
- 有 product_id：有，商品卡和购物车事件都带。
- 有错误码：SSE `error.code` 有；HTTP validation 仍是 FastAPI 默认 422。
- 原生客户端可直接消费：基本可以，Android 有 `SseEventParser` 和 `AgentPayload`。

## 五、RAG 真实性判断

1. 商品数据是否真实入库：是。`product_ingest.py` 从 raw 100 条商品 seed，并生成 products/chunks。
2. 是否真的 embedding：是。`embedding.py` 调 provider embeddings；本次 live smoke 证明 1292 chunks、1024 维。
3. 是否真的向量检索：是，但本次证明的是 SQLite 持久向量路径；pgvector 有代码和 compose，但本次未实跑证明。
4. 是否真的把检索结果注入 LLM：是。`llm_task_payloads.py` 将 criteria、ranked products、evidence、reason atoms 注入 prompt。
5. 是否有结构化过滤：有。类别、预算、品牌、产地、product_type、ingredient/trait avoid。
6. 是否有防幻觉机制：有。候选 ID 校验、商品名范围校验、商业字段拒答、prompt 限制、final decision winner 锁定。
7. 是否可能出现“推荐不存在商品”：商品卡基本不会，因为来自 catalog payload；LLM 自然语言仍需要 sanitize 和 live 测试兜底。
8. 是否能解释每个推荐来自哪条商品数据：能。`product_card.evidence` 带 `source_id/snippet`；本次 live smoke 首条 evidence 为 `p_beauty_011:8`。
