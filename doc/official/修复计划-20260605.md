### 阶段 3：多轮消息历史增强（P1-2）

**目标**：增强 `get_conversation_summary()` 的上下文丰富度

**现状分析**：
- `get_conversation_summary()` 已存在，返回格式化字符串如 "第1轮: 用户'推荐洗面奶', 购买标准'油性肌肤,200元内', 推荐了5个商品"
- 已在 `stages/intent.py` 和 `stages/criteria.py` 中通过 `conversation_context` 参数传给 LLM
- 限制：user_message 截断到 80 字符，只包含 criteria summary 和 product IDs

**我的建议：增强摘要格式，增加 max_turns**

| 文件 | 变更 |
|------|------|
| `backend/src/services/conversation_state.py` | 增强 `get_conversation_summary()`：(1) 增加 max_turns 从 2 到 4；(2) 增加 feedback 信息（用户喜欢的/不喜欢的）；(3) 包含 avoid_products/prefer_traits 等反馈约束 |

**格式示例**：
```
第1轮: 用户'推荐适合油皮的洗面奶，200元以内'
  → 标准: 美妆护肤/油性肌肤/200元内
  → 推荐: 5个商品 (p_beauty_001, p_beauty_002, ...)
  → 用户反馈: 不喜欢 p_beauty_003, 偏好'无酒精'
第2轮: 用户'预算降到150'
  → 标准调整: 预算150元内
  → 推荐: 3个商品
```

**备选方案（不推荐）**：
- 完整消息历史注入：需要 schema 变更，复杂度高
- LLM 压缩摘要：增加延迟和成本

### 阶段 6：clarification prompt 迁移（P2-2）

**目标**：将硬编码的澄清问题迁移到 PromptStore

**现状**：
- `prompts/clarification.md` 已存在，包含完整的 LLM 澄清 prompt
- 但代码中 `slot_checker.py` 使用 `user_messages.py` 中的硬编码字符串
- clarification.md 设计为 LLM 生成澄清问题，但当前是确定性规则

**方案选择**：

选项 A：保留确定性规则，将硬编码字符串移到 clarification.md 的 `## HardcodedQuestions` section
- 优点：保持当前确定性行为，集中管理
- 缺点：PromptStore 加载 .md 文件做字符串查找不够优雅

选项 B：启用 LLM 澄清生成
- 优点：更自然的问题生成
- 缺点：增加 LLM 调用延迟和成本，可能不稳定

**我的建议：选项 A** - 保持确定性行为，将字符串集中到 clarification.md 的一个 section，由 `slot_checker.py` 通过 PromptStore 加载。

| 文件 | 变更 |
|------|------|
| `backend/prompts/clarification.md` | 添加 `## DeterministicQuestions` section，包含 CLARIFY_* 内容 |
| `backend/src/runtime/stages/slot_checker.py` | 改为从 PromptStore 加载问题文本 |
| `backend/src/config/user_messages.py` | 移除 CLARIFY_* 常量 |

### 阶段 7：查询缓存（P2-3）

**目标**：为热门查询添加缓存，减少重复检索

**方案**：基于 criteria hash 的 TTL 缓存

| 文件 | 变更 |
|------|------|
| `backend/src/services/retrieval_cache.py` | 新建，实现 `RetrievalCache`：基于 criteria hash 的 5 分钟 TTL 缓存 |
| `backend/src/services/retriever.py` | `retrieve_with_evidence()` 入口处检查缓存 |
| `backend/src/config/settings.py` | 添加 `RETRIEVAL_CACHE_ENABLED` 和 `RETRIEVAL_CACHE_TTL_SECONDS` 配置 |

**实现要点**：
- 使用 `functools.lru_cache` 或自定义 TTL dict
- 缓存 key：`hash(criteria.model_dump_json())`
- 缓存 value：`RetrievalOutput`（products + evidence + trace）
- 缓存命中时跳过 embedding + pgvector + rerank
- 反馈约束变化时 cache miss（feedback 进入 key）
