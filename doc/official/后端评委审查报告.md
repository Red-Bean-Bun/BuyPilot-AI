# BuyPilot-AI 后端评委审查报告

评审对象：BuyPilot-AI 后端代码  
评审方式：基于代码、配置、接口、测试与 live 运行结果核查，不只采信 README 声明。  
评审日期：2026-05-26

## 一、总评

这个后端已经不是空壳 Demo：FastAPI、SSE、商品入库、chunk embedding、向量召回、rerank、证据绑定、商品卡片、会话状态、反馈、图片上传和购物车状态都有真实代码支撑。最大优点是 RAG 主链路比较完整，且分层基本清楚：`API -> Runtime -> Service -> Repo -> Config/Types`。

最大风险是 live 可靠性还不够稳。`smoke_live_rag` 能证明真实 embedding 与推荐链路可跑通，但 `demo_smoke` 的 6 条路径中，“预算降到200”被真实模型识别成购物车数量更新，导致该轮没有推荐链路。低预算蓝牙耳机、多轮预算收紧、非购物问题等边界输入仍可能触发 `PIPELINE_ERROR` 或返回不满足硬约束的候选。

如果我是评委，我会相信它是真正的 RAG Agent，但会继续追问：哪些路径是稳定工程能力，哪些只是 Demo 场景刚好能跑；当 LLM 误判、rerank/embedding/pgvector 不可用、商品库没有匹配项时，系统是否会诚实失败而不是伪装成功。

## 四、接口与链路核查

核心接口：`POST /chat/stream`，定义于 `backend/src/api/chat.py`。请求体 `ChatStreamRequest` 包含：

- `message: str`，长度 1 到 2000
- `session_id`
- `history`
- `image_url`
- `criteria_patch`
- `client_turn_id`
- `client_trace_id`

响应格式：`text/event-stream`。核心事件：

- `thinking`
- `clarification`
- `criteria_card`
- `text_delta`
- `product_card`
- `cart_action`
- `final_decision`
- `done`
- `error`

商品卡结构：`ProductCardEvent` 包含 `deck_id`、`rank`、`product`、`reason`、`reason_atoms`、`risk_notes`、`evidence`、`actions`。`ProductPayload` 包含 `product_id`、`name`、`price`、`currency`、`image_url`、`category`、`sub_category`、`brand` 等字段。

辅助接口：

- `POST /upload/image`
- `GET /cart/{session_id}`
- `PATCH /cart/{session_id}/items/{product_id}`
- `DELETE /cart/{session_id}/items/{product_id}`
- `POST /feedback`
- `POST /chat/cancel`
- `/health`
- admin eval / observability 相关接口

文字版链路：

```text
用户请求
-> FastAPI /chat/stream
-> runtime.pipeline.chat_stream
-> 图片分析（可选）
-> 意图识别
-> 槽位澄清
-> 购买标准生成
-> query embedding
-> pgvector/SQLite chunk 召回
-> 结构化硬过滤
-> rerank
-> evidence 绑定
-> 推荐文案 / 最终决策
-> SSE 返回 criteria/text/product/final/cart/done
-> conversation / retrieval_trace / evidence_link 持久化
```

## 五、RAG 真实性判断

1. 商品数据是否真实入库：是。raw 数据 100 条，4 品类各 25 条，商品事实来自 `data/raw/ecommerce_agent_dataset`。
2. 是否真的 embedding：是。当前 live smoke 证明 `1292` chunks、`1024` 维 embedding。旧审计里提到的 16 维是历史 deterministic fallback，不应作为当前演示态能力。
3. 是否真的向量检索：是。PostgreSQL 使用 pgvector `<=>`，SQLite 使用持久化 JSON embedding 做 Python cosine scan。
4. 是否真的把检索结果注入 LLM：是。`recommendation_messages` 把 ranked products、evidence context 和 reason atoms 注入 prompt。
5. 是否有结构化过滤：有。包括预算、类目、品牌/产地/成分排除、product_type。
6. 是否有防幻觉机制：有 prompt 约束、候选 ID 校验、商品名校验和库存/优惠/物流等禁词校验，但边界场景仍需加强 no-match 策略。
7. 是否可能出现“推荐不存在商品”：商品卡层面基本不会；自然语言层面已有后验校验，但当 LLM 输出 winner 非候选时仍会导致 error。
8. 是否能解释每个推荐来自哪条商品数据：能。`product_card.evidence.source_id` 指向 `product_chunks`，trace/evidence link 也会持久化。

## 七、本次答辩必问清单

以下清单合并了本次评审发现、Fallback 审计问题和你补充的两组追问。建议答辩前逐题过一遍，能指到具体代码和运行结果。

| # | 评委必问 | 答辩口径 |
|---:|---|---|
| 1 | 你的 chunking 为什么分 `profile / marketing / FAQ / review / risk`？ | `profile` 承载稳定商品属性，保证召回入口；`marketing` 提供官方卖点；`FAQ` 提供官方问答证据；`review` 提供用户体验证据；`risk/warning` 单独标识风险，避免把负面/禁忌内容当正向召回依据。 |
| 2 | chunking 为什么还有 `warning / compare` 这些类型？ | `warning` 用于不适合人群、刺激、过敏等风险提示；`compare` 把同品类比较维度结构化，服务决策卡和多商品比较。它们不是随意切块，而是按导购解释任务切分证据角色。 |
| 3 | pgvector 和 SQLite fallback 的行为差异是什么？ | PostgreSQL 下 `embedding vector(1024)` 走 pgvector `<=>` 和 HNSW 索引，是演示目标；SQLite 下 embedding 以 JSON 保存，用 Python cosine scan，是本地开发兼容路径。SQLite 不是 memory fallback，仍要求 DB 中有真实 chunk embedding。 |
| 4 | pgvector 查询和 SQLite fallback 在排序质量上有什么差异？ | pgvector 可以在库内高效召回 top-k，再接 rerank；SQLite 是全表 Python 余弦扫描，规模小能跑但不代表生产性能。排序质量理论上同 embedding 相似度一致，但工程性能、索引能力和可扩展性差很多。 |
| 5 | 当前 16 维 embedding 是怎么产生的，为什么不是 1024 维？ | 16 维是历史 deterministic fallback，用字符编码生成，目的是开发态不接 provider 也能跑；它不是语义 embedding。当前策略应使用百炼 `text-embedding-v3` 1024 维，演示前必须 `reindex_embeddings` 并用 smoke 检查维度。 |
| 6 | 为什么 live smoke 0 个商品仍然通过？ | 这是旧审计发现的门禁缺陷：只检查 embedding，没有严格断言 product card。当前 smoke 已检查 `product_count >= 1`、evidence、无 error、无 critical fallback；答辩时要强调这是已修过的风险点。 |
| 7 | 为什么 live smoke 有商品仍可能不代表 pgvector 成功？ | 如果用 SQLite dev.db，它证明的是持久化向量后端可用，不证明 pgvector。答辩演示要显式使用 PostgreSQL `DATABASE_URL`，并展示 trace 或 DB 类型。 |
| 8 | 为什么 `recommendation` prompt 没拿到 `evidence_chunks`？ | 当前代码在 system prompt 里传入 `evidence_chunks`，同时 user payload 传 products 和 reason atoms。若评委只看 user JSON，会误以为没传 evidence；要说明 evidence 在 system content 中注入。 |
| 9 | 如何保证推荐理由只引用 evidence？ | prompt 要求优先依据 evidence 与 reason atoms；代码会生成 `reason_atoms` 并绑定 `evidence_id`；LLM 文案还有候选商品名/ID/禁词后验校验。仍需承认自然语言无法 100% 形式化证明，答辩可展示 `product_card.evidence.source_id`。 |
| 10 | 如何保证最终推荐商品一定来自库内？ | `product_card` 只由检索出的 `ProductPayload` 生成，商品来自 raw dataset；`generate_decision` 校验 `winner_product_id` 必须在候选 ID 内。当前缺口是校验失败会报错，建议 fallback 到首候选。 |
| 11 | `洗面奶` 和数据里的 `洁面` 如何匹配？ | 通过 domain terms / product_type alias 归一化，把“洗面奶/洁面乳/洁面”映射到同一产品类型，再和数据 `sub_category` 匹配；同时 chunk alias 里也加入常见用户词增强召回。 |
| 12 | “不要日系品牌”为什么没有被过滤？ | 过滤逻辑有 `origin_avoid`，会把日系映射到 SK-II、资生堂等品牌并排除。若没有过滤，通常是 intent/criteria 阶段没抽出 `origin_avoid`；需要对“不日系/不要日本品牌”加确定性抽取或回归测试。 |
| 13 | 用户问优惠券/库存时为什么不会编造？ | prompt 禁止编造优惠、库存、物流、购买承诺；`llm_client` 对推荐文案做禁词校验，发现“库存/优惠券/满减/包邮”等会失败。更稳的做法是专门 handler 回复“商品库没有该字段，不能确认”。 |
| 14 | 多轮澄清状态存在数据库哪里？ | `conversations` 表保存 `session_id`、`criteria_json`、`product_ids`、`user_message`；`conversation_state` 读取最近 criteria 和商品 ID，用于下一轮标准合并和指代解析。 |
| 15 | 多轮里“这个”“第二个商品”如何解析？ | runtime 保存上一轮 `product_ids`；“这个/这款/刚才”默认指向上一轮第一个商品；“第二个/第2款”通过 ordinal 解析成下标 1。相关逻辑在 `runtime.handlers._referenced_product_id`。 |
| 16 | “把这个加到购物车”的“这个”如何解析，上一轮为空时怎么办？ | 若 intent 没给 `target_product_id`，后端从上一轮 `product_ids` 取；上一轮为空时返回 clarification，不会凭空加购。当前加购成功依赖上一轮推荐已持久化。 |
| 17 | rerank 失败时排序依据是什么？ | 当前严谨策略是不把 rerank 失败伪装成成功，provider 不可用会抛 `RerankUnavailable`。如果要做降级，应明确记录 fallback，并用 `filter_score + vector_score + price` 的确定性排序，而不是静默替换。 |
| 18 | 图片上传后如何影响检索，而不只是拼到 prompt？ | `/upload/image` 保存图片；`analyze_image` 用 VLM 输出 `category_hint/description/visible_traits`；pipeline 将其转成 `category/product_type/skin_type/dietary/use_scenario` 等检索条件，再进入 criteria 和 retriever。 |
| 19 | 图片上传后 VLM 结果如何转成检索条件？ | `pipeline._image_analysis_to_retrieval_constraints` 从 VLM 文本提取品类、产品类型、肤质、无糖/低糖、日常场景等，并写入用户消息上下文，criteria LLM 再结构化到 `Constraints`。 |
| 20 | pgvector 和 SQLite fallback 各自什么时候允许使用？ | 日常开发允许 SQLite JSON embedding；答辩/演示必须用 PostgreSQL + pgvector。fallback 可以存在，但不能在关键依赖失效时仍声称“真实 pgvector RAG 成功”。 |
| 21 | 哪些模块是核心手写设计，哪些是 demo fallback？ | 核心手写设计：SSE schema、pipeline stage、retriever、chunking、evidence、conversation/cart state、LLM task interface、trace/audit。demo/fallback：旧 `mock_pipeline.py`、测试里的 external AI mock、SQLite dev backend、历史 16 维 deterministic embedding。 |
| 22 | 哪些代码是为了比赛评审权重主动设计的，而不是 AI 拼接？ | 评审权重驱动的设计包括：结构化商品卡、evidence source_id、防幻觉后验校验、SSE event schema、live smoke 门禁、retrieval trace、购物车 CRUD、图片转检索条件。答辩要按“为什么对评分有价值”解释。 |
| 23 | 为什么测试通过不等于真实链路通过？ | 单测默认 mock external AI，验证 schema、事件顺序、过滤、状态；真实 provider、embedding、rerank、pgvector 要靠 `smoke_live_rag` 和 `demo_smoke`。所以必须同时展示 pytest 和 live smoke。 |
| 24 | 如果商品库没有合适商品，系统应该怎么做？ | 正确行为是返回 no-match/clarification，说明哪些约束导致无结果，而不是放宽硬约束后发不满足条件的卡片。当前这是优先修复点。 |
| 25 | 当前最危险的 live 稳定性问题是什么？ | “预算降到200”会被误判成购物车数量更新；低预算蓝牙耳机会放宽预算并在 decision 阶段报错；非购物问题会进入推荐链路。这三类会被评委随手测到。 |

## 八、最终结论

结论：

- 是否达到合格标准：接近但有风险
- 是否达到优秀标准：否，部分工程结构达到优秀，但 live 稳定性没有达到
- 如果明天答辩，最应该先修的 3 件事是：
  1. 修复“预算降到200”多轮路径，确保 `demo_smoke` 6/6
  2. 禁止无匹配时发违反预算/品类/product_type 的商品卡
  3. 修复 CI mypy 和 decision winner fallback，避免评委现场看到 `PIPELINE_ERROR`
