• 一、总评

  这个后端已经不像 mock 项目，核心 RAG 链路是真实存在的：PostgreSQL/pgvector、商品 chunk embedding、检索、rerank、
  证据绑定、SSE 商品卡都能在本机 live smoke 跑通。我会相信它是一个真正的 RAG Agent，而不是 README 包装出来的
  demo。最大优点是分层清楚，API -> Runtime -> Service -> Repo 基本落实，RAG 证据和商品卡结构也足够支撑 Android 渲
  染。最大风险是交付一致性：全量测试仍有 2 个契约失败，demo_smoke 实测挂起超过 10 分钟无输出后被我终止。答辩最可能
  被追问的是：为什么 live smoke 通过但完整 demo smoke 不稳定，以及前端 golden trace 为什么和当前后端事件语义不一
  致。

  三、硬伤清单

  P0：

  - demo_smoke 实测超过 10 分钟无输出，我已终止。位置：backend/src/scripts/demo_smoke.py:184。建议给每个 turn 加
    asyncio.wait_for，并逐场景流式打印状态。

  - 全量测试失败 2 项，均来自 golden trace 语义不一致。位置：backend/tests/test_viewmodel_contract.py:359、
    contracts/examples/demo_budget_beauty.sse:29。建议按当前 PRD 拆成两个 turn，或按 turn 分组校验 seq。

  P1：

  - README 仍写“默认 SQLite / 日常开发 SQLite”，但 settings 无 DATABASE_URL 会直接退出，runtime SQLite 也被拒绝。
    位置：README.md:14、backend/src/config/settings.py:106。建议统一为“Postgres 必需，SQLite 仅 pytest”。

  - 流式推荐文本直接透传 LLM delta，非流式才做未知商品/优惠/库存校验。位置：backend/src/services/
    llm_client.py:155。建议流结束后做缓冲校验，或只让 LLM 输出非事实过渡文案。

  - /cart view 聊天事件只返回 cart_action=view，没有购物车明细。位置：backend/src/runtime/handlers.py:100。建议返
    回结构化 cart summary 或让客户端明确调用 /cart/{session_id}。



  - 无 .dockerignore，构建上下文可能携带 .venv/cache/uploads。建议补上。
  - 本地 .env 有真实 key/token 设置，但 .env 未被 git 跟踪且 .gitignore 已忽略；当前未发现 tracked key 泄漏。

  四、接口与链路核查

  核心链路：POST /chat/stream -> backend/src/api/chat.py:14 -> backend/src/runtime/pipeline.py:92 -> intent/
  criteria/retrieval/recommendation/decision -> SSE。
  返回事件包括 thinking / clarification / criteria_card / text_delta / product_card / cart_action /
  final_decision / done / error，schema 在 backend/src/types/sse_events.py:221。
  商品卡含 product_id/name/price/image_url/category/brand/reason/evidence/actions，可被原生客户端消费。
  session_id、turn_id、seq、event_id 都有；空字符串和超长输入由 Pydantic 限制在 backend/src/types/schemas.py:20。

  五、RAG 真实性判断

  1. 商品数据真实入库：是，backend/src/services/product_ingest.py:34 读取 100 条 raw JSON。
  2. embedding：是，backend/src/services/embedding.py:35 调 provider；live smoke 结果为 1292 个 1024 维 chunk。
  3. 向量检索：是，backend/src/repos/documents.py:60 使用 pgvector <=>。
  4. 检索结果注入 LLM：是，backend/src/services/llm_task_payloads.py:113 传入 products/evidence/reason atoms。
  5. 结构化过滤：是，backend/src/services/retriever.py:326 做品类、预算、品牌、产地、子类、成分排除。
  6. 防幻觉：有，但不完美；商品卡来自库内，决策 winner 被校验，商业字段有拦截；流式自然语言仍依赖 prompt。
  7. 推荐不存在商品：商品卡层面风险低；文本层仍有小风险。
  8. 可解释来源：是，evidence.source_id 指向 chunk，live smoke 首条证据为 p_beauty_011:8。
