# BuyPilot-AI 后端评委审查总评（2026-06-01）

审查日期：2026-06-01  
审查对象：基于 RAG 的多模态电商智能导购 AI Agent 后端，兼看接口是否支撑 Android 端到端交互。  
审查依据：`doc/official/评委审查指令.md` 与 `doc/official/课题说明会-基于RAG的多模态电商智能导购AI-Agent.md`。  
审查方法：以代码、契约、测试和可复现命令为准；README 和状态文档只作为线索。  
本地验证：`timeout 900s backend/.venv/bin/python -m pytest backend/tests -q`，结果 `285 passed, 4 warnings in 227.74s`。未执行 live smoke，原因是真实 RAG smoke 依赖外部网络、真实模型 Key 和可能的云调用费用。

## 一、总评

这个后端已经达到可参赛交付的形态，不是简单 mock 或套壳 demo：代码里有真实商品数据读取、chunking、embedding、pgvector 检索、rerank、证据绑定、SSE 流式事件、会话状态和购物车持久化。最大优点是 RAG 真实性和工程边界清楚，`API -> Runtime -> Service -> Repo -> Config/Types` 的职责基本守住，商品卡片也能带 `product_id`、价格、图片、证据和推荐理由。最大风险不在主代码能力，而在答辩验收可复现性：`.env.example` 的 `[REQUIRED] KEY=value` 不是普通 dotenv 写法，且 `smoke_live_rag.py` 仍要求同轮出现 `final_decision`，与当前 product-first 两阶段决策流程不完全一致。作为评委，我会相信它是真做了 RAG Agent，但会追问 live smoke 是否在当前提交、当前配置、当前模型 Key 下实际跑通。效果可靠性已有较多测试兜底，但真实 LLM 输出质量、prompt injection 严格性、多模态“相似图检索”深度还不能仅凭本地 pytest 证明。答辩时最可能被追问的是：RAG 检索证据如何绑定到最终文案、为什么候选商品阶段不立即给最终决策、以及文档宣称的模型主备策略与 `llm_profiles.yaml` 当前配置为何不一致。

## 三、硬伤清单

### P0：会导致 Demo 失败或严重扣分的问题

1. `.env.example` 的必填变量写法会破坏普通 dotenv 复制流程  
   代码位置：`.env.example:22`、`.env.example:72`；加载逻辑见 `backend/src/config/settings.py`。  
   问题描述：模板使用 `[REQUIRED] DATABASE_URL=...` 和 `[REQUIRED] BAILIAN_API_KEY=...`，复制为 `.env` 后，项目自带 `_load_env_file()` 会把 key 解析成 `[REQUIRED] DATABASE_URL`，而不是 `DATABASE_URL`。  
   为什么违反课题要求：课题强调可低成本启动；评委按 README `cp .env.example .env` 后填值，后端可能仍认为关键变量缺失。  
   建议修复：改为合法 dotenv，例如 `DATABASE_URL=...`、`BAILIAN_API_KEY=`，用注释标注 required。  
   修复优先级：P0。

2. `smoke_live_rag.py` 与当前 product-first 两阶段决策流程不一致  
   代码位置：`backend/src/scripts/smoke_live_rag.py:109`；当前流程见 `backend/src/runtime/handlers.py:508` 到 `backend/src/runtime/handlers.py:545`。  
   问题描述：脚本仍把 `"final_decision" in tags` 作为单轮 chat 成功条件；当前多候选推荐设计是首轮 `product_card + text_delta + criteria_card + done(awaiting_product_feedback)`，用户再说“继续”才出 `final_decision`。  
   为什么违反课题要求：README 把 live smoke 作为评委门禁，脚本若误报失败，会直接影响答辩可信度。  
   建议修复：把 smoke 改为两轮：第一轮断言商品卡和证据，第二轮发送“继续”断言 `final_decision`；或只在单候选场景断言同轮决策。  
   修复优先级：P0。

### P1：会明显影响评分的问题

3. 模型策略文档与运行配置不一致  
   代码位置：`CLAUDE.md:21` 到 `CLAUDE.md:26`、`README.md` 技术决策表；实际配置见 `backend/src/config/llm_profiles.yaml:1` 到 `backend/src/config/llm_profiles.yaml:13`。  
   问题描述：文档说 Doubao 做意图主力、Qwen-Plus 做生成主力；实际 profile 把 `analyze_intent/generate_criteria/generate_recommendation/generate_decision` 主力都设为 `qwen_turbo`，Doubao 是 fallback。  
   为什么违反课题要求：评委会按代码追问模型选型，文档和配置不一致会削弱“能解释原理”的可信度。  
   建议修复：要么把 YAML 改回文档策略，要么更新文档说明当前选择 qwen-turbo 的原因和质量/成本取舍。  
   修复优先级：P1。

4. 多模态能力是 VLM 属性抽取，不是视觉相似检索  
   代码位置：`backend/src/runtime/stages/multimodal.py:1` 到 `backend/src/runtime/stages/multimodal.py:8`、`backend/src/runtime/message_rules.py:75` 到 `backend/src/runtime/message_rules.py:113`。  
   问题描述：图片分析结果拼进文本消息参与 criteria 和 RAG 检索，没有 image embedding 或图片-商品相似召回。  
   为什么违反课题要求：课题加分项“拍照找货”考察非文本输入转化为检索语义；当前可支撑“图片辅助导购”，但不宜包装成完整相似图检索。  
   建议修复：答辩表述为“VLM 提取属性驱动 RAG”；若冲多模态高分，补一个可验证的 image-to-product matching 路径。  
   修复优先级：P1。

5. full pytest 虽通过，但存在异步资源清理 warning  
   代码位置：测试输出显示 `tests/test_state_repos.py::test_pipeline_followup_restores_criteria_from_database` 触发 `aiosqlite` 线程在 event loop 关闭后回调。  
   问题描述：不影响当前测试结果，但说明 SQLite 测试隔离路径存在连接生命周期瑕疵。  
   为什么违反课题要求：工程质量维度会看错误处理和资源管理；warning 过多会影响评委对稳定性的判断。  
   建议修复：在相关 fixture 中显式 await engine dispose/connection close，避免 loop close 后后台线程回调。  
   修复优先级：P1。

### P2：可以优化但不致命的问题

6. `data/processed/products.json` 是空文件  
   代码位置：`data/processed/products.json`。  
   问题描述：运行时主要读取 `data/raw/ecommerce_agent_dataset`，但 processed 产物为空，容易让审查者误以为数据处理链断裂。  
   为什么违反课题要求：数据治理是课题维度之一，派生产物不一致会带来观感风险。  
   建议修复：要么删除不用的 processed 空文件，要么重生成并在 README 说明 raw 是唯一事实源。  
   修复优先级：P2。

7. golden trace 仍有旧同轮 `final_decision` 示例  
   代码位置：`contracts/examples/demo_budget_beauty.sse`。  
   问题描述：示例里 product deck 后直接同轮 final_decision，与当前 product-first 多候选两阶段流程不一致。  
   为什么违反课题要求：前后端契约示例是评委和客户端最容易看的材料，过期示例会造成误解。  
   建议修复：拆成两个 trace：首轮候选 deck，次轮“继续”决策。  
   修复优先级：P2。

8. 工程性能优化加分项相对弱  
   代码位置：`backend/src/runtime/handlers.py:333` 到 `backend/src/runtime/handlers.py:386` 已有投机检索并行，但没有热门查询缓存。  
   问题描述：当前性能亮点主要是投机检索、心跳和流式分块，不是缓存或首 token 量化优化。  
   为什么违反课题要求：4.4 是加分项，不做不致命，但不能在答辩中主打性能深度。  
   建议修复：如时间允许，加一个低风险的 retrieval/criteria stage timing 指标展示；缓存不建议临近答辩硬加。  
   修复优先级：P2。

文字版链路：

用户请求  
-> `backend/src/api/chat.py:62` 的 `POST /chat/stream`  
-> `backend/src/runtime/pipeline.py` 注册 turn、取消 token、审计事件  
-> 如有图片，`runtime/stages/multimodal.py` 调 `services.llm_client.analyze_image`，再把图片分析拼入用户消息  
-> `runtime/stages/intent.py` 调 `services.llm_client.analyze_intent`，注入会话摘要  
-> `runtime/stages/slot_checker.py` 做必要澄清，例如手机缺预算  
-> `runtime/stages/criteria.py` 调 `generate_criteria` 或合并 `criteria_patch`  
-> `runtime/handlers.py` 并行启动投机检索与标准生成  
-> `services.retriever.retrieve_with_evidence()` 调 embedding、pgvector recall、硬过滤、rerank、证据选择  
-> `runtime/handlers.py` 先发 `product_card`，多候选时再流式生成推荐解释文本，随后发 `criteria_card` 和 `done(awaiting_product_feedback)`  
-> 用户继续/反馈后，`continue_decision_from_current_deck()` 读取上一轮 deck、feedback、候选商品，运行显式评分并让 LLM 解释，输出 `final_decision`  
-> Android 依据 SSE event type 渲染文本、候选商品 deck、标准卡、购物车事件和最终决策卡。

架构识别：

- 后端框架：FastAPI。
- 向量数据库：PostgreSQL + pgvector；SQLite 仅测试隔离使用 JSON embedding 兼容。
- Embedding：`backend/src/services/embedding.py`，profile `qwen_embedding/text-embedding-v3/1024`。
- Rerank：`backend/src/services/reranker.py`，profile `gte_rerank/qwen3-rerank`。
- LLM 网关：`backend/src/services/llm_client.py` 任务接口，底层 `backend/src/services/llm_gateway.py`。
- RAG 核心：`backend/src/services/retriever.py:76` 的 `retrieve_with_evidence()`。
- 商品 schema：raw JSON 由 `backend/src/repos/products.py` 读取，DB 表见 `backend/src/repos/models.py` 的 `Product` 和 `ProductChunk`。
- 商品卡 schema：`backend/src/types/sse_events.py:96` 的 `ProductPayload` 与 `backend/src/types/sse_events.py:169` 的 `ProductCardEvent`。
- 会话状态：`conversations` 保存 criteria、deck_id、product_ids；feedbacks/cart_items 按 session 维护状态。

## 五、RAG 真实性判断

1. 商品数据是否真实入库？  
   是。仓库内 raw 商品 JSON 数量为 100；`seed_products()` 从 raw dataset 生成 `Product` 和 `ProductChunk`，见 `backend/src/services/product_ingest.py:40` 到 `backend/src/services/product_ingest.py:116`。

2. 是否真的 embedding？  
   是。入库时 `_embed_chunks()` 调 `embed_texts()`；查询时 `retrieve_with_evidence()` 调 `embed_text()` 生成 query embedding，见 `backend/src/services/retriever.py:81` 到 `backend/src/services/retriever.py:83`。

3. 是否真的向量检索？  
   是。PostgreSQL 路径由 `list_vector_chunks_by_similarity()` 执行 pgvector similarity search；retriever 再做硬过滤和 rerank。

4. 是否真的把检索结果注入 LLM？  
   是。`recommendation_stream_messages()` 和 `decision_messages()` 都把 `ranked_products`、`evidence_chunks`、`reason_atoms_by_product` 注入 prompt，见 `backend/src/services/llm_task_payloads.py:147` 到 `backend/src/services/llm_task_payloads.py:179`。

5. 是否有结构化过滤？  
   有。硬过滤覆盖 category、budget、brand_avoid、origin_avoid、product_type、ingredient_avoid、feedback avoid_products/avoid_traits，见 `backend/src/services/retriever.py` 的 `_FILTER_CHECKS` 相关逻辑。

6. 是否有防幻觉机制？  
   有。商品卡只来自检索候选；推荐文案校验未知商品 ID、候选外商品名和库存/优惠/物流等禁词；商业 claim 问题有确定性拦截，见 `backend/src/runtime/message_rules.py:22` 到 `backend/src/runtime/message_rules.py:50`。

7. 是否可能出现“推荐不存在商品”？  
   商品卡层面风险低；LLM 文本层面有校验，但最终决策说明仍依赖模型解释和清洗函数，不能说完全无风险。

8. 是否能解释每个推荐来自哪条商品数据？  
   基本能。`ProductCardEvent` 带 `evidence`，evidence 有 `source_id` 和 snippet；retrieval trace/evidence link 表用于审计。

## 八、最终结论
