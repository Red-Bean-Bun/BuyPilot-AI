# BuyPilot-AI 答辩演示脚本

> 3 分钟核心链路演示，按顺序走 4 条 Demo 路径。每条 30-45 秒。
> 评分维度映射：基础功能 35% + 工程质量 25% + 效果可靠性 20% + 加分项深度 20%

---

## 路径 1：模糊推荐 + 条件筛选（45 秒）
**输入**："推荐适合油皮的洗面奶，200元以内"

**评委话术**：
"第一条路径演示核心推荐链路。用户输入模糊需求'推荐适合油皮的洗面奶，200元以内'。系统首先做意图识别——我们用百炼 Qwen-Turbo 主力，Doubao 兜底，意图识别准确率 90%。然后自动生成结构化购买标准——品类、肤质、预算、场景——这里可以看到 criteria_card 事件。接着触发混合检索：BM25 关键词召回 + pgvector 语义召回，用 RRF 算法融合排序，再经 qwen3-rerank 精排。最后流式返回推荐文案和商品卡片，每张卡片带证据链接，点击可查看推荐依据来自商品库的哪条 chunk。整条链路从输入到首张商品卡约 1.2 秒。"

**评分维度**：基础功能完整性（端到端链路）、效果可靠性（证据绑定、无幻觉）
**SSE 事件序列**：thinking → criteria_card → text_delta → product_card × N → final_decision → done

## 路径 2：拍照找货 + 多模态理解（40 秒）
**输入**：上传护肤品图片 + "这个适合敏感肌吗？"

**评委话术**：
"第二条路径演示多模态——拍照找货能力。用户上传一张护肤品照片，问'这个适合敏感肌吗？'。图片经 multipart 上传后，Qwen-VL-Plus 做视觉理解，识别品类和成分特征。同时我们的图片 embedding 索引——用 qwen3-vl-embedding 预建了 100 张商品图的 1024 维向量——做视觉相似检索。文本和图像共享同一向量空间，支持以图搜图和以文搜图无缝切换。VL 理解结果注入购买标准生成，最终返回匹配敏感肌的商品推荐。这是加分项 4.2 多模态交互 ⭐⭐⭐ 档。"

**评分维度**：多模态交互（⭐⭐⭐ 拍照找货）、对话智能（VL 理解 + RAG 融合）
**SSE 事件序列**：thinking → criteria_card → product_card × N → text_delta → done

## 路径 3：多轮反问 + 反选排除（45 秒）
**输入**："不要含酒精的防晒霜" → "预算降到200"

**评委话术**：
"第三条路径演示多轮对话和反选排除。第一轮用户说'不要含酒精的防晒霜'，系统需要理解否定语义——这不只是关键词过滤，'不含酒精但含烟酰胺'这种复杂约束需要 LLM 理解否定作用域。我们用确定性规则层做毫秒级否定词抽取，LLM 做语义深度理解，两层协同。第二轮用户追加'预算降到200'，系统自动合并历史约束——防晒品类、不含酒精、预算 200 以内——渐进收敛推荐结果。约束满足率 89%，反选过滤准确率 100%。这是加分项 4.3 对话智能 ⭐⭐ 档。"

**评分维度**：对话智能（⭐⭐ 反选排除）、多轮上下文管理、效果可靠性
**SSE 事件序列**：（轮1）thinking → criteria_card → product_card × N → done → （轮2）thinking → criteria_card（合并） → product_card × N → final_decision → done

## 路径 4：对话式购物车 + 下单（50 秒）
**输入**："把第一个加到购物车" → "查看购物车" → "结算"

**评委话术**：
"第四条路径演示业务闭环——对话式购物车。用户说'把第一个加到购物车'，系统识别 add_to_cart 意图，通过 cart_action SSE 事件通知客户端更新购物车状态。整个购物车 CRUD——加购、改数量、删除——都通过自然语言驱动，持久化到 PostgreSQL cart_items 表。最后结算，系统检查购物车非空后生成 checkout_preview，展示订单汇总。注意这里不做真实支付——按照赛题指引我们聚焦对话式操作结构化数据的闭环能力。这是加分项 4.1 业务闭环 ⭐ 入门档。"

**评分维度**：业务闭环（⭐ 对话式加购）、工程质量（cart_items 持久化、cart_action 事件）
**SSE 事件序列**：thinking → cart_action(add) → done → thinking → cart_action(view) → done → thinking → cart_action(checkout) → done

## 技术亮点收尾（备用 20 秒）

如果评委追问技术深度：

1. **BM25 + RRF 混合检索**："召回层用 BM25 做精确关键词匹配，pgvector 做语义召回，RRF 算法融合两路排名——它只依赖排名而非分数值，从根本上规避向量相似度和 BM25 分数量纲不同的问题。对精确品牌查询，召回率提升 40%+。"

2. **三端协议守卫**："SSE 事件协议是封闭 DSL——10 种事件类型。我们有 Python import-time、Kotlin build-time、CI script 三层自动化守卫。协议漂移 = 程序无法启动或编译，不是'测试可能发现'。"

3. **618 单元测试**："618 个测试覆盖 model/schema/pipeline/API/retrieval/state repos/observability/reliability 全部层级。纯函数如 `_passes_hard_filters` 至少有 3 个直接测试用例。"

4. **评测闭环**："20 条样本 × 15 指标，包含确定性指标（约束满足、证据覆盖）和 LLM Judge 指标。综合分 78.4%，faithfulness 从 55% 提升到 82%。`make eval` 一键复现。"

## 常见评委追问预案

| 问题 | 回答要点 |
|------|---------|
| "为什么用百炼主力而不是 Doubao？" | Qwen-Turbo 在结构化 JSON 输出上更稳定，延迟更低（1.2s vs Doubao 5-15s）。Doubao 作为 fallback 保证可靠性 |
| "怎么保证不编造商品信息？" | 价格/库存/SKU 从 PostgreSQL 直查，LLM 只生成推荐文案。商业禁词检测 + 价格白名单校验。所有推荐理由绑证据 source_id |
| "为什么不做真实支付？" | 赛题指引明确：⭐ 入门档聚焦对话式操作结构化数据。真实支付需要地址/订单状态机，不符合 3 周时间约束 |
| "数据只有 100 条，够用吗？" | 赛题重点考察工程能力。100 条 × 4 品类 × 1292 semantic chunk 足以验证 RAG 链路正确性。数据质量比数量重要 |
