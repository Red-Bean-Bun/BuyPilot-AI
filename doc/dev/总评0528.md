# BuyPilot-AI 后端评委审查总评（0528）

审查对象：基于 RAG 的多模态电商智能导购 AI Agent 后端  
审查方法：按 `doc/official/评委审查指令.md`，以代码、契约、测试和本地数据库状态为准；README 仅作线索。  
本地验证：`timeout 900s backend/.venv/bin/python -m pytest -q backend/tests`，结果 `171 passed in 86.42s`。  
补充核查：本地 SQLite `backend/buypilot-dev.db` 含 `products=100`、`product_chunks=1292`、`embedded_chunks=1292`，抽样 embedding 维度为 1024。未执行真实云模型 live smoke，因为它依赖外部网络、真实 API Key 和可能的云调用费用；本报告对 live 能力按代码和脚本可验证性评分。

## 一、总评

这个后端已经不像“套壳 demo”，而是一个有真实分层、真实商品库、真实 chunk embedding、向量召回、硬过滤、rerank、SSE 事件契约和状态表的 RAG Agent 后端。最大优点是工程结构和可审计性较强：`API -> Runtime -> Service -> Repo -> Config/Types` 边界清楚，商品卡片带 `product_id`、价格、图片、证据片段和理由原子，测试覆盖也比较扎实。最大风险是当前主链路的交互语义和 demo/smoke 期望不一致：默认 `/chat/stream` 只到购买标准确认，`auto_run=true` 只返回商品卡，不返回 `final_decision`，而 `demo_smoke.py` 和 `smoke_live_rag.py` 都期待单轮出现商品卡和最终决策。作为评委，我会相信它是真正做了 RAG 检索和结构化返回，但不会直接相信 README 中“Demo Smoke 6/6 通过”的说法，因为当前代码与 smoke 脚本自相矛盾。答辩时最可能被追问的是：为什么端到端默认链路不直接产出推荐商品和最终决策、为什么推荐正文 LLM streaming 函数没有接入主推荐路径、以及 live smoke 是否真的在当前提交跑通过。

## 三、硬伤清单

### P0：会导致 Demo 失败或严重扣分的问题

1. 默认 `/chat/stream` 不走完整推荐链路  
   代码位置：`backend/src/runtime/handlers.py:229`、`backend/src/runtime/handlers.py:260`、`backend/src/runtime/handlers.py:630`  
   问题描述：首次请求默认只返回 `criteria_card` 和确认文本，随后 `done.finish_reason=awaiting_criteria_confirmation`，不会检索、不会返回商品卡。  
   为什么违反课题要求：评委输入“推荐适合油皮的洗面奶，200元以内”时，期望端到端看到商品推荐和卡片，而不是必须知道隐藏参数 `auto_run=true`。  
   建议修复：明确产品交互。如果比赛 Demo 要一轮出结果，默认开启推荐阶段；如果坚持确认式交互，README、前端契约、demo smoke 都必须改成两步，并提供一键 demo 脚本。  
   修复优先级：P0。

2. `auto_run=true` 只返回商品卡，不返回最终决策，但 smoke 脚本要求 `final_decision`  
   代码位置：`backend/src/runtime/handlers.py:331`、`backend/src/runtime/handlers.py:336`、`backend/src/runtime/handlers.py:337`；脚本位置：`backend/src/scripts/demo_smoke.py:61`、`backend/src/scripts/demo_smoke.py:65`、`backend/src/scripts/smoke_live_rag.py:80`、`backend/src/scripts/smoke_live_rag.py:84`  
   问题描述：当前推荐阶段在商品卡后以 `awaiting_product_feedback` 结束；`final_decision` 只有下一轮“继续”才出现。但 demo/smoke 脚本的第一轮直接期待 `final_decision=True`。  
   为什么违反课题要求：答辩前 smoke gate 会误报失败，或者 README 声称通过但评委无法复现。  
   建议修复：二选一：把 smoke 改成“请求 auto_run -> 收商品卡 -> 再发继续 -> 收 final_decision”，或在 `auto_run` 模式下自动执行 decision。  
   修复优先级：P0。

3. 推荐解释 LLM streaming 已实现但未接入主推荐路径  
   代码位置：`backend/src/services/llm_client.py:155`、`backend/src/runtime/handlers.py:494`、`backend/src/runtime/handlers.py:331`  
   问题描述：`stream_recommendation()` 和 `_stream_recommendation_text_events()` 存在，但 `continue_recommendation_from_criteria()` 直接发商品卡，没有调用推荐正文生成；用户可见文本主要是固定的标准分析文案。  
   为什么违反课题要求：课题要求“通过 RAG 生成推荐理由、流式返回 AI 回复”。商品卡理由是确定性 reason atom，质量可控，但不能证明推荐正文真的由 RAG prompt 生成并流式输出。  
   建议修复：检索后先调用 `ctx.stages.run_recommendation_text_stream(criteria, products, evidences)`，把 text_delta 与 product_card 组合输出；保留当前 reason atoms 作为卡片事实约束。  
   修复优先级：P0。

### P1：会明显影响评分的问题

4. Eval runner 默认不会进入推荐阶段，评测指标可能失真  
   代码位置：`backend/src/runtime/eval_runner.py:213`、`backend/src/runtime/eval_runner.py:223`  
   问题描述：评测时构造 `ChatStreamRequest(message=question, ...)`，没有设置 `auto_run=true`，因此多数样本只采到 criteria，而采不到商品卡和推荐答案。  
   为什么违反课题要求：如果答辩展示评测模块，指标不能真实反映检索和推荐能力。  
   建议修复：评测 runner 使用与 Demo 一致的两步或 auto-run 流程，并把最终 `product_card`、`text_delta`、`final_decision` 纳入指标。  
   修复优先级：P1。

5. 前后端契约文档与实现不完全一致  
   代码位置：`contracts/frontend-integration.md:121`、`contracts/frontend-integration.md:142`、`backend/src/api/upload.py:37`  
   问题描述：契约文档仍称 cancel 是 stub、上传支持 JSON 兼容；代码实际 cancel 已持久化 best-effort，上传只接受 multipart，否则 415。  
   为什么违反课题要求：原生客户端联调按文档实现会失败，评委也会质疑文档可信度。  
   建议修复：以 `backend/src/types/schemas.py` 和实际路由为准刷新 `contracts/frontend-integration.md`，删除不支持的 JSON 上传描述。  
   修复优先级：P1。

6. 多模态是“图片理解 -> 文本线索 -> 检索”，还不是严格的相似图检索  
   代码位置：`backend/src/runtime/stages/multimodal.py:8`、`backend/src/runtime/message_rules.py:74`、`backend/src/runtime/message_rules.py:93`  
   问题描述：图片上传和 VLM 分析存在，分析结果拼进 message 参与后续 criteria，但没有图片 embedding 或视觉相似商品召回。  
   为什么违反课题要求：可以支撑“图片辅助问答/找货”，但如果宣称“拍照找相似商品”会被追问深度。  
   建议修复：答辩表述为“VLM 提取可见属性驱动 RAG 检索”；若要拿多模态高分，增加 image-to-product matching 或基于图片商品 ID 的样例闭环。  
   修复优先级：P1。

7. 商品卡片没有跳转链接和 SKU 细节  
   代码位置：`backend/src/types/sse_events.py:95`、`backend/src/repos/products.py:83`  
   问题描述：`ProductPayload` 有 `product_id/name/price/image_url/category/brand`，但无 `product_url`、SKU 列表或库存字段。  
   为什么违反课题要求：基础卡片可渲染，但购物闭环和具体规格选择能力有限。  
   建议修复：如果前端需要“跳转信息”，在 `ProductPayload` 加 `sku_options` 或 `product_url`；没有真实字段时明确置空并禁止 LLM 编造。  
   修复优先级：P1。

### P2：可以优化但不致命的问题

8. `data/processed/products.json` 中存在个人机器绝对路径  
   代码位置：`data/processed/products.json:10` 等多处  
   问题描述：派生数据里保存了 `/mnt/disk1/LZJ/project/BuyPilot-AI/...` 绝对路径。运行时主要读 `data/raw`，但仓库审查会看到个人路径。  
   为什么违反课题要求：影响可移植性观感，可能让评委怀疑 demo 依赖个人机器。  
   建议修复：重生成 processed 文件，图片路径统一为相对路径或 `/assets/products/...`。  
   修复优先级：P2。

9. 依赖文件是 `pyproject.toml + uv.lock`，但 README 中“依赖版本锁定”需要解释  
   代码位置：`backend/pyproject.toml:10`、`backend/uv.lock`  
   问题描述：`pyproject.toml` 使用 `>=` 范围，锁定实际依赖靠 `uv.lock`。  
   为什么违反课题要求：评委若只看 `pyproject` 会以为没有固定版本。  
   建议修复：README 明确“以 uv.lock 锁版本”，Dockerfile/CI 使用 `uv sync --locked`。  
   修复优先级：P2。

10. 评测样本 ground_truth 字段与当前 Constraints DSL 不完全一致  
    代码位置：`data/eval/eval_samples.json`、`backend/src/services/eval/metrics.py:16`

    为什么违反课题要求：评测指标会低估或错判真实约束抽取。  
    建议修复：把 eval ground truth 迁移到当前 `Constraints` 字段，并保留别名映射。  
    修复优先级：P2。

### 文字版链路还原

用户请求  
-> `backend/src/api/chat.py:14` 的 `POST /chat/stream`  
-> `backend/src/runtime/pipeline.py` 创建 `session_id/turn_id`、注册取消 token、审计 turn  
-> 如有 `image_url`，`runtime/stages/multimodal.py` 调 `services.llm_client.analyze_image`，再由 `message_rules.message_with_image_context()` 拼入用户消息  
-> `runtime/stages/intent.py` 调 `services.llm_client.analyze_intent`，注入最近会话摘要  
-> `slot_checker.py` 判断是否缺品类、预算、商品类型，必要时返回 `clarification`  
-> `runtime/stages/criteria.py` 调 `generate_criteria` 或合并 `criteria_patch`，生成 `CriteriaPayload`  
-> 默认停在 `criteria_card`；若 `auto_run=true` 或特殊 patch，则进入检索  
-> `runtime/stages/recommendation.py` 调 `services.retriever.retrieve_with_evidence`  
-> `services.retriever` 调 `embedding.embed_text` 生成查询向量，从 PostgreSQL pgvector 或 SQLite JSON embedding chunk 召回，应用硬过滤和 feedback 过滤，再调用 `reranker.rerank_texts`  
-> `runtime/handlers.py` 生成 `product_card`，卡片含商品结构化字段、理由原子和 evidence  
-> 后续用户说“继续”时，`handle_continue()` 基于上一轮 product_ids 调 `runtime/stages/decision.py` -> `generate_decision()` 输出 `final_decision`  
-> SSE 按 `types/sse_events.py` 格式返回给 Android。

### 架构识别

- 后端框架：FastAPI。
- 向量数据库：演示环境 PostgreSQL + pgvector；本地开发 SQLite 用 JSON embedding + Python cosine 兼容。
- Embedding 调用：`backend/src/services/embedding.py:35`，使用 profile `qwen_embedding/text-embedding-v3/1024`。
- LLM 调用封装：`backend/src/services/llm_client.py` 任务接口，底层 `backend/src/services/llm_gateway.py` OpenAI-compatible chat completions。
- RAG 核心文件和函数：`backend/src/services/retriever.py:69` 的 `retrieve_with_evidence()`，`backend/src/repos/documents.py` 的 chunk 读取和 pgvector 查询。
- 商品数据 schema：raw JSON 含 `product_id/title/brand/category/sub_category/base_price/image_path/skus/rag_knowledge`；DB 表见 `backend/src/repos/models.py` 的 `Product` 和 `ProductChunk`。
- 商品卡片 schema：`backend/src/types/sse_events.py:95` 的 `ProductPayload` 和 `backend/src/types/sse_events.py:167` 的 `ProductCardEvent`。
- 流式协议：SSE，`format_sse()` 输出 `event:` + `data:`。
- 多轮状态：`conversations` 表保存 criteria、deck_id、product_ids；feedbacks/cart_items 按 session 保存。
- 购物车闭环：支持 chat intent 加购/移除/改数量/查看，也有 REST CRUD。
- 多模态：支持图片上传、VLM 分析、图片线索转检索条件；不是视觉相似向量检索。

## 五、RAG 真实性判断

1. 商品数据是否真实入库？  
   是。`seed_products()` 从 `list_raw_products()` 读取官方 raw JSON，写入 `Product` 和 `ProductChunk`，见 `backend/src/services/product_ingest.py:34`。本地 DB 统计为 100 个商品、1292 个 chunk。

2. 是否真的 embedding？  
   是。入库时 `_embed_chunks()` 批量调用 `embed_texts()`，见 `backend/src/services/product_ingest.py:41`；查询时 `retrieve_with_evidence()` 调 `embed_text()`，见 `backend/src/services/retriever.py:75`。本地 DB 抽样 embedding 维度为 1024。

3. 是否真的向量检索？  
   是。PostgreSQL 下用 `embedding <=> CAST(:query_embedding AS vector)`；SQLite 下读取持久化 embedding 做 cosine similarity，见 `backend/src/services/retriever.py:123` 和 `backend/src/repos/documents.py:63`。

4. 是否真的把检索结果注入 LLM？  
   部分是。`recommendation_messages()` 和 `decision_messages()` 会把 `ranked_products`、`evidence_chunks` 和 `reason_atoms` 注入 prompt，见 `backend/src/services/llm_task_payloads.py:112`。但当前主推荐路径未调用 recommendation text streaming，只在后续 decision 阶段调用 LLM。

5. 是否有结构化过滤？  
   有。硬过滤覆盖 session feedback、category、budget、brand_avoid、origin_avoid、product_type、ingredient_avoid，见 `backend/src/services/retriever.py:360` 到 `backend/src/services/retriever.py:424`。

6. 是否有防幻觉机制？  
   有但不完整。商品卡来自库内 `ProductPayload`，推荐文案校验未知商品 ID、候选外商品名、库存/优惠/物流等禁词，见 `backend/src/services/llm_client.py:122`；商业字段问题有确定性拦截，见 `backend/src/runtime/message_rules.py:19`。不足是主路径推荐正文未接入，prompt injection 的语义防护仍主要依赖 prompt 和候选校验。

7. 是否可能出现“推荐不存在商品”？  
   商品卡层面风险低，因为 `product_card.product` 来自检索候选和 raw dataset。LLM 文本层面有校验函数，但当前主路径没有正式输出 LLM 推荐正文；decision 的 summary/why 虽校验 winner_id，但没有同等严格过滤候选外商品名和商业 claims。

8. 是否能解释每个推荐来自哪条商品数据？  
   基本能。`product_card.evidence` 带 `source_id` 和 snippet；`trace_recorder` 保存 retrieval trace/evidence links；reason atom 带 `evidence_id`。这对答辩解释 RAG 来源是加分项。

## 六、建议作者优先修复的 10 件事

1. 统一 demo 主链路：让默认 `/chat/stream` 一轮返回 `criteria_card + text_delta + product_card`，或正式把两步确认写进 README、前端契约和 demo 脚本。
2. 修复 `demo_smoke.py` 和 `smoke_live_rag.py`：按当前真实流程传 `auto_run=true`，并在商品卡后再发“继续”验证 `final_decision`；不要单轮期待不可能出现的事件。
3. 把 `run_recommendation_text_stream()` 接进 `continue_recommendation_from_criteria()`，让 RAG 推荐正文真实流式输出，而不是只有固定 criteria 文案和商品卡理由。

## 八、最终结论

结论：
- 是否达到合格标准：是，但有 demo 风险。
- 是否达到优秀标准：部分达到。RAG 工程和可审计性接近优秀，但端到端默认体验、smoke 一致性和推荐正文流式生成还没闭合。
- 如果明天答辩，最应该先修的 3 件事是：
  1. 修复 `/chat/stream`、`demo_smoke.py`、`smoke_live_rag.py` 的流程不一致，让一键 smoke 真实通过。
  2. 把 RAG 推荐正文 streaming 接入主路径，并保留商品卡 reason atoms 作为事实约束。
  3. 更新前后端契约和 README，明确 `auto_run`、两步决策、上传方式、live provider 前置条件和可复现验收命令。
