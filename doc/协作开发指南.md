# BuyPilot-AI 三人协作开发指南

> 写给刚加入的两位开发者：一人前端（Android），一人后端主开发（FastAPI），一人后端算法/检索。前端 1 人，后端 2 人。

---

## 一、环境搭建（所有人，5 分钟）

### 前置条件

- 已安装 Git、Docker、Docker Compose
- 有百炼平台 API Key（找负责人要）

### 克隆 & 启动

```bash
git clone <仓库地址> BuyPilot-AI
cd BuyPilot-AI

# 复制环境变量（后端开发需要）
cd backend
cp .env.example .env
# 编辑 .env，填入百炼 API Key

# 启动开发环境
cd ../deploy
docker compose up -d
```

Docker Compose 会拉起 PostgreSQL（含 pgvector）容器。FastAPI 用本地 venv 开发，不跑在容器里。

### 本地开发依赖安装（后端）

```bash
cd backend
python -m venv .venv
source .venv/bin/activate
pip install -e .
```

### 验证环境

```bash
curl http://localhost:8000/health  # 应返回 {"status": "ok"}
psql -h localhost -U buypilot -d buypilot  # 密码: buypilot
```

## 三、接口契约（第一天就定好，不要改）

### 3.1 POST /chat/stream

**请求体**：
```json
{
  "message": "给4岁孩子买室内益智玩具，预算200，不要小零件",
  "session_id": null,
  "image_url": null,
  "history": [],
  "criteria_patch": null,
  "skip_stages": []
}
```

**SSE 事件流**（7 种，按顺序到达）：

| 事件 | 触发时机 | 前端动作 |
|------|---------|---------|
| `thinking` | 收到请求立即返回 | 更新 inline thinking 气泡（不是顶部状态栏） |
| `clarification` | 缺少必填槽位（如年龄） | 显示问题卡片 |
| `criteria_card` | 购买标准生成完毕 | 渲染标准卡 + quick_actions |
| `text_delta` | 推荐文本流式输出 | 追加到同一条 AI 气泡 |
| `product_card` | 每个商品推荐就绪 | 插入商品卡片 |
| `final_decision` | 最终结论就绪 | 渲染决策卡 |
| `done` | 全部完成 | 恢复输入框 |

### 3.2 POST /upload/image

- **请求**：`multipart/form-data`，字段名 `file`
- **响应**：`{"image_url": "...", "width": 1280, "height": 960}`

### 3.3 POST /feedback

- **请求**：`{"session_id": "...", "product_id": "...", "action": "like|dislike|view_detail|click_alternative", "reason": "..."}`
- **响应**：`{"status": "ok"}`

## 四、前端开发者指引（Android 负责人）

### 你只需要关注

1. **SSE 连接**：用 OkHttp SSE 连接后端 `/chat/stream`
2. **ChatUiNode**：用 `sealed interface` 定义所有 UI 节点类型
3. **LazyColumn**：统一承载文本消息和卡片的时间线
4. **输入区**：文本 + Photo Picker 图片选择 + 停止生成
5. **session_id 管理**：首次请求 session_id 为 null，从首个 SSE 事件中提取后端生成的 session_id 并缓存
6. **停止生成**：关闭 SSE 连接 + best-effort 调用 `/chat/cancel`

### 开工顺序

| 天 | 任务 |
|---|------|
| Day 1 | 新建 Android 项目（Kotlin + Compose），写好 `sealed interface ChatUiNode` 和所有 `data class` |
| Day 2 | 实现 OkHttp SSE 连接 + 解析 7 种事件 |
| Day 3 | LazyColumn + 文本气泡 + thinking 状态 |
| Day 4 | 三类卡片组件（criteria_card / product_card / final_decision） |
| Day 5 | 输入区 + 图片上传 + 停止生成 |
| Day 6-7 | 联调 + 反馈按钮 + Debug 页 |

### 必读文档

- `doc/prd/01-Android前端PRD.md`（完整 PRD，含 Kotlin data class 定义、SSE 事件示例、卡片 UI 规范）
- `doc/decisions/决策记录.md`（第 11 条 sealed interface、第 13 条图片上传两步走）

## 五、后端主开发指引（FastAPI 负责人）

### 你只需要关注

1. **SSE 管道编排**：`backend/src/runtime/pipeline.py`，用 async generator 串联各 stage
2. **API 路由**：`backend/src/api/` 下按端点分文件
3. **数据模型**：`backend/src/types/schemas.py` + `backend/src/types/sse_events.py`

5. **session_id 生成**：首次请求 session_id 为 null 时生成新 session_id，在所有事件 envelope 中返回
6. **反馈注入**：通过 session_id 查询 feedbacks 表获取历史反馈，注入 criteria 约束

## 七、Git 协作规范

### 分支策略

```
master          ← 稳定分支，只有 Demo 就绪时才合并
├── feat/android-chat     ← 前端
├── feat/api-pipeline     ← 后端主开发
└── feat/retrieval-rag    ← 后端算法
```

- 每人开自己的 feature 分支，不要碰别人的
- 每天至少 push 一次，不要让代码烂在本地
- 合并到 master 前必须：本地 `docker compose up` 跑通主链路

### Commit 规范

```
feat: 实现 OkHttp SSE 连接和 ChatUiNode sealed interface
fix: 修复 product_card 渲染时 LazyColumn 抖动
docs: 更新 Android PRD SSE 事件示例
```

类型：`feat` / `fix` / `docs` / `refactor` / `test`

## 八、联调方式

### 第一阶段（Day 1-3）：各干各的

- 前端：Mock SSE 数据，先跑通 UI 渲染
- 后端：用 `curl` 或 Python 脚本测试 API，不依赖前端

### 第二阶段（Day 4-5）：点对点联调

- 前端 OkHttp SSE 连后端本地 `localhost:8000`
- 验证 7 种 SSE 事件都能正确渲染
- 图片上传 + 多模态链路

### 第三阶段（Day 6-7）：端到端 Demo

- 三条 Demo 路径全部跑通
- 反馈闭环（点"不喜欢" → 下一轮结果改变）

1. **先看文档**：`doc/prd/` + `doc/decisions/` 里有答案
