## 一、RAG 检索链路

### 1.1 检索流程中硬过滤、向量召回、rerank 的执行顺序是什么？为什么这样排？

先 SQL 硬过滤（品类/预算/肤质/品牌/产地等确定性约束）→ pgvector 1024 维语义召回 top-k → qwen3-rerank 精排。硬过滤在最前面是因为确定性约束不应该被语义相似度"绕过"——用户说"200元以内"，就不该召回 800 元的商品。

### 1.2 检索结果为空怎么办？

检索松弛策略：只放宽预算（按 `CHEAPER_BUDGET_RATIO` 上浮），不放宽品类。如果松弛后仍无结果，返回 `_no_match_followup_text()` 动态建议（放宽预算/换品类/去掉排除条件），不会硬凑一个不满足条件的商品。

### 1.3 严格过滤无结果时为什么可以放宽预算？

预算是连续值，用户说"200元以内"通常有弹性；但品类是离散的，放宽品类等于推荐用户不想要的东西。所以松弛只动预算。代码在 `retriever.py` 的 `_relaxation_attempts` 中定义。

### 1.4 rerank 失败时排序依据是什么？

当前策略是不把 rerank 失败伪装成成功，provider 不可用会抛 `RerankUnavailable`。如果做降级，应用 `filter_score + vector_score + price` 的确定性排序，并明确记录 fallback，而不是静默替换。

## 二、Chunking 策略

### 2.1 1292 个 chunk 是怎么从 100 个商品切出来的？

每个商品 JSON 含 `rag_knowledge`：marketing_description + official_faq[] + user_reviews[]。按语义角色切分——marketing_description 是一个 chunk，每个 FAQ 是一个 chunk，每个 review 是一个 chunk。100 个商品 × 平均 ~13 个 chunk = 1292。

### 2.2 为什么按 profile/marketing/FAQ/review/warning/compare 语义切分而不是固定长度？

按角色切分让每个 chunk 的证据角色明确：profile 承载稳定商品属性（召回入口）；marketing 提供官方卖点；FAQ 提供官方问答证据；review 提供用户体验证据；warning 单独标识风险（避免把负面内容当正向召回依据）；compare 把同品类比较维度结构化，服务决策卡。固定长度切分会破坏证据的语义完整性。

## 三、向量库与 Embedding

### 3.1 为什么选 pgvector 而不是 Chroma 或 Qdrant？

pgvector 与 PostgreSQL 同库部署，零额外运维；100 条商品的规模不需要专用向量数据库的分布式能力；HNSW 索引足够覆盖演示需求。减少一个基础设施依赖 = 减少一个出故障的点。

### 3.2 pgvector 和 SQLite fallback 的行为差异是什么？

PostgreSQL 下 embedding 走 `vector(1024)` + pgvector `<=>` 算子 + HNSW 索引，是演示目标。SQLite 下 embedding 以 JSON 保存，用 Python cosine scan，是本地开发兼容路径。日常开发允许 SQLite；答辩/演示必须用 PostgreSQL + pgvector。

### 3.3 当前 16 维 embedding 是怎么产生的，为什么不是 1024 维？

16 维是历史 deterministic fallback，用字符编码生成，目的是开发态不接 provider 也能跑；它不是语义 embedding。演示前必须 `reindex_embeddings` 用百炼 `text-embedding-v3` 生成 1024 维，并用 `smoke_live_rag` 检查维度。

### 3.4 HNSW 索引参数是怎么调的？

100 条商品规模下默认参数即可。`m=16, ef_construction=64` 是 pgvector 推荐值，对于这个数据量不需要特殊调优。

## 四、防幻觉

### 4.1 如何保证推荐理由只引用 evidence？

三层防御：(1) prompt 要求优先依据 evidence 与 reason_atoms；(2) 代码生成 `reason_atoms` 并绑定 `evidence_id`，每个理由可溯源到具体 chunk；(3) LLM 文案有禁词后验校验，发现"库存/优惠券/满减/包邮"等编造词会失败。

### 4.2 如何保证最终推荐商品一定来自库内？

`product_card` 只由检索出的 `ProductPayload` 生成，商品来自 raw dataset。`generate_decision` 校验 `winner_product_id` 必须在候选 ID 内。LLM 没有机会改写商品卡的价格、图片、商品名字段——这些字段直接从数据库读取。

### 4.3 用户问优惠券/库存时为什么不会编造？

prompt 禁止编造优惠、库存、物流、购买承诺；`llm_client` 对推荐文案做禁词校验。更稳的做法是专门 handler 回复"商品库没有该字段，不能确认"（已实现：`_emit_commercial_claim_reply`）。

### 4.4 prompt injection 要求"忽略商品库直接编 5 个商品"时，怎么防护？

商品卡只由检索结果生成，不受 LLM 文本输出影响。即使 LLM 被注入，product_card 的 product_id/name/price/image_url 来自数据库，不在 LLM 控制范围内。text_delta 可能有幻觉文案，但禁词校验会拦截。

### 4.5 reason_atoms 是什么设计？为什么不直接让 LLM 生成推荐理由文本？

`reason_atoms` 是结构化的理由单元，每个含 `dimension`（如 skin_type/budget）+ `value` + `evidence_id`。好处：(1) 前端可按维度渲染证据标签；(2) 每个理由可点击跳转到 evidence chunk 原文；(3) LLM 只负责自然语言润色，事实由结构化数据保证。

## 五、多轮上下文

### 5.1 多轮怎么记住上一轮？

`conversations` 表保存 `session_id`、`criteria_json`、`product_ids`、`user_message`。`conversation_state` 读取最近 criteria 和商品 ID，下一轮 criteria LLM 接收 `existing` criteria 做合并，列表字段（brand_avoid/ingredient_avoid）累积去重，hard constraint 字段从历史继承。

### 5.2 "这个""第二个商品"如何解析？

`runtime/cart_rules.py` 的 `referenced_product_id()`：保存上一轮 product_ids；"这个/这款/刚才"默认指向上一轮第一个商品；"第二个/第2款"通过序数词解析成下标。上一轮为空时返回 clarification，不会凭空操作。

### 5.3 第二轮"要轻量的"是怎么关联到第一轮的"跑鞋"的？criteria 合并逻辑是什么？

意图阶段拿到 `extracted_constraints`；criteria 阶段调 `generate_criteria()` 传入 `existing` criteria（上一轮保存的），LLM prompt 里写"在它的基础上修改"，把上一轮的 category/product_type 继承下来，只修改本轮新增的约束。

## 六、购物车与指代解析

### 6.1 "把第二个加到购物车""删除第二款""改成 2 件"如何解析指代和持久化状态？

Android 端 `resolveAddToCartTarget()` 做模糊匹配：序数词提取 + 品牌/品类/名称多维打分 + 歧义检测。后端购物车状态存在 `cart_items` 表（session_id + product_id + quantity），跨轮通过 session_id 关联。

### 6.2 购物车状态存在哪里，如何跨轮引用？

`cart_items` 表，按 session_id 隔离。每轮 cart_action 事件返回完整 cart snapshot（items/total_items/total_price），前端直接渲染，不需要额外查询。

### 6.3 上一轮推荐为空时用户说"加到购物车"怎么办？

`referenced_product_id()` 返回 None → handler 发 clarification 事件问"你想把哪个商品加入购物车？"，不会凭空加购。

## 七、多模态

### 7.1 图片能力到底是视觉相似检索，还是 VLM 属性抽取后走文本 RAG？

双通道：(1) VLM 属性理解（qwen-vl-plus 识别品类/属性/肤质等 → 转成检索约束）；(2) image embedding 视觉召回（qwen3-vl-embedding 生成 1024 维视觉向量 → pgvector 相似度匹配）。两个通道并行，结果合并。不是单纯的图片描述。

### 7.2 图片上传后 VLM 结果如何转成检索条件？

`pipeline._image_analysis_to_retrieval_constraints` 从 VLM 文本提取品类、产品类型、肤质、无糖/低糖、日常场景等，写入用户消息上下文，criteria LLM 再结构化到 Constraints。

## 八、SSE 协议与架构

### 8.1 SSE 的 text_delta 和 product_card 是并行发出的吗？如果 LLM 流式输出中断，已发出的商品卡怎么处理？

不是并行。顺序是：thinking → text_delta(开场白) → product_card × N → criteria_card → text_delta(推荐文案) → final_decision → done。商品卡来自检索结果，不依赖 LLM 流式输出；即使 LLM 中断，已发出的商品卡仍然有效。

### 8.2 前后端协议怎么保证一致？

三层守卫：(1) `contracts/sse-events.schema.json` 是 source of truth；(2) Python 端 `sse_events.py` import-time 校验 schema 对齐；(3) Kotlin 端 assemble 强制跑契约测试。新增 event type 必须三端同步更新。

### 8.3 Android 如何解析 product_card/final_decision/cart_action？

`SseEventParser.kt` 解析 SSE data JSON → `AgentPayload.kt` sealed class 区分事件类型 → `ChatUiNode.kt` 按类型渲染对应 Composable 组件。

## 九、模型与工程决策

### 9.1 为什么用 qwen-turbo 做生成主力，而不是 qwen-plus？

task-oriented interface 按任务选模型。意图/标准/推荐/决策用 qwen-turbo（延迟低、成本低），结构化 prompt + JSON schema 约束下效果与 qwen-plus 无显著差异。多模态用 qwen-vl-plus，embedding 用 text-embedding-v3。配置见 `llm_profiles.yaml`。

### 9.2 开发态和生产态怎么区分？

`STRICT_RUNTIME=1` 禁用确定性 fallback（16 维 embedding、mock rerank），暴露 live 失败。`ALLOW_MEMORY_STATE_FALLBACK=1` 只适合开发态。答辩演示用 strict 模式，证明不依赖 mock。

### 9.3 哪些测试是真链路，哪些只是 mock 链路？

`pytest` 默认 mock external AI，验证 schema、事件顺序、过滤逻辑、状态管理。`smoke_live_rag` 和 `demo_smoke` 用真实 provider + 真实 pgvector，验证端到端链路。答辩要同时展示两者。

### 9.4 哪些代码是核心手写设计，哪些是 demo fallback？

核心手写：SSE schema、pipeline stage、retriever、chunking、evidence、conversation/cart state、LLM task interface、trace/audit。demo/fallback：旧 mock_pipeline.py、测试里的 external AI mock、SQLite dev backend、历史 16 维 deterministic embedding。

### 9.5 哪些代码是为了比赛评审权重主动设计的，而不是 AI 拼接？

评审权重驱动的设计：结构化商品卡 + evidence source_id（35% 基础功能）、防幻觉后验校验（减分项防御）、SSE event schema 三端对齐（25% 工程质量）、live smoke 门禁（20% 可靠性）、购物车 CRUD（20% 加分项）。

## 十、评委可能随手测的边界场景

### 10.1 "帮我选"被误分类怎么办？

Convergence 协议：前端发 `converge=true` 强制走 continue handler，跳过 LLM 意图分类。

### 10.2 "今天天气怎么样"会不会进入推荐链路？

意图阶段识别为 chitchat，返回引导文案"我是购物助手，可以帮你找商品、比价格、加购物车"，不进检索。

### 10.3 "有没有戴森的吹风机"（库中没有的商品）？

`_unsupported_product_type()` 拦截，返回"当前商品库暂不覆盖吹风机，我不能基于现有数据给出可靠推荐"。不编造、不硬凑。

### 10.4 重复查询怎么处理？

检索结果有 TTL 缓存（`retrieval_cache.py`），相同 criteria hash 命中缓存直接返回，减少重复 embedding + rerank 调用。

## 十一、课题说明会官方 Q&A

### 11.1 必须 iOS 和 Android 都做吗？

不需要。选择 iOS 或 Android 任一平台即可，重点是实现原生体验，不接受纯 Web/H5 方案。

### 11.2 50-100 条数据够吗？RAG 效果会不会很差？

本课题重点考察工程能力而非数据量。50-100 条足以验证 RAG 链路的正确性，建议精心设计数据覆盖不同类目和场景。

### 11.3 可以使用 AI 开发吗？

可以。鼓励使用 AI 开发，但需要主导架构设计并了解代码实现原理和细节。答辩时需能解释链路和代码中的细节。

### 11.4 可以用 LangChain / LlamaIndex 等框架吗？

可以。但需要理解框架内部原理。
