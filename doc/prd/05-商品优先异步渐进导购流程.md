# 商品优先异步渐进导购流程

> 日期：2026-05-28  
> 状态：最新产品与前后端协作决策  
> 背景：对齐最新版课题说明会中“端到端链路是否跑通：客户端对话 → 后端 RAG 检索 → 模型生成 → 流式返回 → 商品卡片展示”的 P0 要求，同时保留 BuyPilot 的结构化筛选、SwipeDeck 反馈和最终决策收敛体验。

## 1. 决策摘要

我们不再把 `criteria_card` 放在商品推荐前面作为默认确认步骤。

新的核心思路：

```text
先让用户看到商品，再让用户基于商品理解和调整筛选标准。
```

也就是：

```text
由：先 criteria_card，再 product_card
改为：先 product_card，再 criteria_card 作为筛选调整卡
```

`criteria_card` 不取消，但它的角色从“前置确认卡”变成“后置筛选调整卡”。

## 2. 当前问题

当前实现更接近“三段式确认型决策助手”：

```text
第 1 步：用户说需求
→ 后端生成 criteria_card
→ 停下来等用户确认购买标准

第 2 步：用户确认 / 继续
→ 后端开始检索商品
→ 返回 product_card
→ 前端聚合成 ProductDeckNode / ProductRecommendationStrip

第 3 步：用户左滑 / 右滑 / 反馈
→ 后端结合反馈
→ 返回 final_decision
```

这个产品逻辑成立，但有 P0 风险：用户说“推荐适合油皮的洗面奶，200 元以内”后，第一轮只看到标准卡，看不到商品卡。评委可能认为基础推荐链路没有跑完。

最新版课题说明会的最小闭环强调：

```text
用户对话 → 后端 RAG 检索 → 模型生成 → 流式返回 → 商品卡片展示
```

因此基础 Demo 默认必须出现 `product_card`，前端必须能展示可点击商品卡片。

## 3. 新流程

改成“先推荐，再筛选调整”的异步渐进式导购：

```text
用户说需求
→ 后端理解需求
→ 如果信息足够，直接检索商品
→ 返回 2-5 个 product_card
→ 前端聚合成 ProductDeckNode / ProductRecommendationStrip
→ 用户先看到真实候选商品
→ 后端随后展示 criteria_card
→ criteria_card 用来解释当前筛选范围，并提供快捷调整入口
→ 用户可点卡调整，也可直接用自然语言补充要求
→ 用户可 swipe / 反馈 / 加购
→ 需要时再生成 final_decision
```

一句话：

```text
由：标准确认优先
改为：商品结果优先，标准卡辅助调整
```

## 4. 设计理由

### 4.1 符合评委 P0

官方基础 Demo 要看到商品推荐结果：

```text
用户对话 → RAG 检索 → 流式返回 → 商品卡片展示
```

因此默认第一轮必须有 `product_card` / `ProductDeckNode`。不能要求前端或用户知道 `auto_run=true` 这种隐藏开关。

### 4.2 用户体验更直接

用户说出需求后，最想先看到“有什么可选商品”。如果先让用户确认一堆标准，会像表单，会打断购买兴致。

### 4.3 用户常常不知道该筛什么

例如用户只说“200 元以内洗面奶”，他可能并不知道还要看：

```text
肤质
清洁力
是否含酒精
氨基酸 / 皂基
敏感肌风险
日常 / 深层清洁
```

先展示几张商品卡后，再给 `criteria_card`，用户更容易理解“原来还可以从这些维度筛”。

### 4.4 criteria_card 变成导购筛选器

`criteria_card` 不再是拦路确认，而是类似淘宝、京东筛选器的结构化调整入口。

但它不能替代自然语言调整。最好的体验是“双入口”：

```text
用户可以点筛选卡调整
也可以直接说“再温和一点”“不要酒精”“预算再低点”
```

`criteria_card` 的价值是给用户一个“我还能从哪些维度调整”的参考面板，而不是把用户锁进表单。

## 5. 卡片角色定义

### 5.1 clarification / 澄清卡

事件名：

```text
clarification
```

前端节点/组件：

```text
ClarificationNode / ClarificationBlock
```

触发条件：

```text
只在信息太模糊，无法进入有效检索时出现。
```

示例：

```text
推荐东西
推荐一款商品
我想买点生活用品
```

这类输入缺少品类或关键入口信息，系统必须先问。

### 5.2 product_card / 商品卡

事件名：

```text
product_card
```

前端聚合节点/组件：

```text
ProductDeckNode / ProductRecommendationStrip
```

角色：

```text
默认优先出现，满足基础推荐闭环。
```

后端可以连续发 2-5 条 `product_card`。前端按同一个 `deck_id` 聚合成一个 `ProductDeckNode`，聊天流里看到的是“候选商品卡组”。用户点进去后，进入全屏 `ProductSwipeModeScreen`，每个商品才是一张可左滑 / 右滑的 swipe card。

### 5.3 criteria_card / 筛选调整卡

事件名：

```text
criteria_card
```

前端节点/组件：

```text
CriteriaNode / CriteriaSummaryCard / CriteriaEditSheet
```

新角色：

```text
后置出现，用来解释当前推荐范围，并提供筛选调整入口。
```

它展示当前系统理解出的筛选条件，例如：

```text
品类：美妆护肤
商品类型：洁面
预算：200 元以内
适用对象：油性肌肤
场景：日常护肤
可调整项：更温和 / 不含酒精 / 控油更强 / 敏感肌适用
```

它不再表达“请确认后我才推荐”，而是表达“你可以用这些维度继续筛”。

`criteria_card` 不是唯一调整方式。用户仍然可以直接在输入框里说：

```text
再温和一点
不要酒精
预算再低点
想要控油更强的
敏感肌也能用的
```

这些自然语言调整应被后端解析为新的 criteria patch，并触发新一轮推荐。

### 5.3.1 criteria_card 视觉与信息结构改进

后置 `criteria_card` 应从“购买标准确认表”改成“当前筛选器”。

建议文案：

```text
标题：当前筛选
副标题：已按这些条件找到候选，可继续优化
主按钮：调整筛选
辅助提示：也可以直接说“再温和一点”“不要酒精”
```

建议信息层级：

```text
第一层：当前硬条件
例如：洁面、200 元以内、油性肌肤

第二层：推荐优化方向
例如：温和度、清洁力、成分避雷、肤质适配

第三层：快捷调整
例如：不要酒精、再温和一点、控油强一点、预算降低
```

这样它不会像“确认表单”，而像一个可操作的导购筛选面板。

### 5.4 final_decision / 最终决策卡

事件名：

```text
final_decision
```

前端节点/组件：

```text
FinalDecisionNode / DecisionSummaryCard
```

角色：

```text
用户看过商品、滑过卡、调整过标准后，再生成。
```

它回答：

```text
综合你的选择，最终更推荐哪一个，为什么，有什么风险。
```

## 6. 后端改动

### 6.1 默认不再等待 criteria 确认

`/chat/stream` 的基础推荐默认不再停在：

```text
done(awaiting_criteria_confirmation)
```

`ChatStreamRequest.auto_run` 不应再作为前端必须知道的隐藏开关。可选实现方式：

```text
方案 A：删除默认等待逻辑，让普通推荐默认继续检索。
方案 B：保留 auto_run 字段，但默认行为等价于 auto_run=true。
方案 C：新增 confirm_before_recommendation=true，仅在显式请求时启用旧确认流。
```

推荐采用方案 A 或 C。

### 6.2 默认推荐事件流

目标事件流：

```text
thinking(understanding)
→ thinking(retrieval)
→ product_card(rank=1, deck_id=xxx)
→ product_card(rank=2, deck_id=xxx)
→ product_card(rank=3, deck_id=xxx)
→ criteria_card
→ done(awaiting_product_feedback, deck_id=xxx)
```

为了让用户更快看到结果，`product_card` 应先于 `criteria_card` 出现。`criteria_card` 可在商品卡后紧接着发，也可以在检索完成后以筛选调整卡形式发出。

### 6.3 criteria_card 语义调整

后端仍然需要生成 criteria，因为它用于：

```text
检索 query 构造
硬过滤
推荐理由解释
前端筛选调整卡展示
criteria_patch 重跑推荐
```

但对外文案和 quick actions 需要从“确认购买标准”调整为“优化筛选范围”。

不再默认发送类似：

```text
请确认这些购买标准，确认后我再推荐。
```

应改为：

```text
我先按这些条件筛了一组候选，你也可以继续调整范围。
```

同时后端必须继续支持自然语言调整。用户输入：

```text
再温和一点
不要酒精
预算再低点
```

时，后端应结合上一轮 `criteria` 和当前 `deck_id`，解析为结构化约束变更，例如：

```text
ingredient_avoid += ["酒精"]
preference += ["温和"]
budget_max 下调
```

然后触发新一轮检索，而不是只依赖用户点击 `criteria_card`。

### 6.4 clarification 触发边界

只有真正缺少关键入口信息时才发 `clarification`。

建议规则：

```text
缺 category 且无法从文本/图片推断 → clarification
品类价格跨度极大且用户完全没有预算或偏好 → 可 clarification
已有明确商品类型、预算、肤质、用途、反选条件中的任一强约束 → 优先检索
```

示例：

```text
“推荐适合油皮的洗面奶，200元以内” → 不澄清，直接检索
“200元以下的蓝牙耳机有哪些？” → 不澄清，直接检索
“推荐护肤品” → 可先澄清，也可返回宽泛候选后给筛选卡
“推荐一款商品” → 澄清 category
```

### 6.5 criteria_patch 重跑推荐

用户通过后置 `criteria_card` 调整筛选条件，或直接用自然语言补充要求后：

```text
criteria_patch / 新一轮 message
→ 后端合并上一轮 criteria
→ 生成新 criteria
→ 新 deck_id
→ 重新检索
→ 返回新的 product_card deck
→ 返回新的 criteria_card
→ done(awaiting_product_feedback, deck_id=新 deck)
```

旧 deck 在前端可保留历史状态，但新 deck 是当前有效候选。

自然语言调整和卡片调整应进入同一条后端路径：

```text
自然语言调整 → intent/criteria 解析 → criteria patch → 新 deck
卡片调整 → explicit criteria_patch → 新 deck
```

两者只是输入方式不同，最终都修改结构化 criteria。

### 6.6 final_decision 收敛

候选阶段不要同轮自动输出 `final_decision`。候选阶段停在：

```text
done(awaiting_product_feedback, deck_id=xxx)
```

用户执行以下操作后，再触发收敛：

```text
左滑 / 右滑
打开详情
查看证据
点击“帮我选一个”
输入“继续 / 收敛建议 / 给最终建议”
```

收敛阶段读取：

```text
session_id
deck_id
product_ids
feedbacks
criteria
```

然后输出：

```text
thinking(decision)
→ final_decision
→ done(completed)
```

### 6.7 后端测试调整

需要修改或新增测试：

```text
基础推荐默认出现 product_card
基础推荐默认不以 awaiting_criteria_confirmation 结束
product_card 先于 criteria_card 或至少同轮出现
done 为 awaiting_product_feedback 且带 deck_id
clarification 只在缺关键入口信息时出现
criteria_patch 产生新的 deck_id 并返回新 product_card
final_decision 只在收敛阶段出现
```

旧测试中断言“默认不出现 product_card”的用例，需要改为显式确认模式测试，或删除。

## 7. 前端改动

### 7.1 不再把 criteria_card 当作前置确认

前端收到 `criteria_card` 后，仍正常渲染 `CriteriaSummaryCard`，但它不再代表“推荐前确认”。它代表“当前推荐范围”和“筛选调整入口”。

文案建议从：

```text
修改标准
```

调整为：

```text
调整筛选
优化推荐范围
换一组条件
```

### 7.2 product_card 是第一优先展示内容

前端收到 `product_card` 后，继续沿用现有逻辑：

```text
按 deck_id 聚合 product_card
生成 ProductDeckNode
渲染 ProductRecommendationStrip
```

聊天流里优先让用户看到候选商品卡组。`ProductRecommendationStrip` 是基础 Demo 的关键可见证据。

### 7.3 商品候选卡组与 Swipe 模式

聊天流：

```text
ProductRecommendationStrip
```

展示：

```text
候选商品
N 个
横向预览图
商品名 / 价格 / 标签
```

点击后进入：

```text
ProductSwipeModeScreen
```

支持：

```text
左滑：not_interested
右滑：like
打开详情：view_detail
查看证据：open_evidence
加入购物车：add_to_cart
```

这些行为继续通过 feedback API 或已有 action 记录。

### 7.4 后置 criteria_card 展示方式

`criteria_card` 在商品卡组后出现时，前端应避免让它看起来像“之前漏掉的一步确认”。它应该更像一个筛选器：

```text
当前按这些条件筛选
可继续优化推荐范围
```

建议 UI 文案：

```text
当前筛选
已按这些条件找到候选
调整筛选
也可以直接说“再温和一点”“不要酒精”
```

底板仍使用 `CriteriaEditSheet`，但交互语义是“调整并重新推荐”。

### 7.5 自然语言调整仍是一等入口

前端不能把用户引导成“只能点卡片调整”。输入框始终可用，且 placeholder / 微文案可以暗示用户可直接补充：

```text
可以继续说：再温和一点、不要酒精、预算低一点
```

用户输入这类调整时，前端正常发 `/chat/stream`，不需要先打开筛选卡。后端负责把自然语言补充合并到上一轮 criteria。

### 7.6 处理事件顺序

前端需要支持以下顺序：

```text
product_card* → criteria_card
```

也要继续兼容历史顺序：

```text
criteria_card → product_card*
```

因为后端迁移期间或 mock 模式可能存在旧序列。

### 7.7 loading 与状态

收到第一条 `product_card` 后，用户已经看到了基础结果。此时：

```text
可以保留轻量 loading，等待更多商品或 criteria_card
不要阻塞用户打开商品
不要要求用户先处理 criteria_card
```

收到：

```text
done(awaiting_product_feedback, deck_id=xxx)
```

前端停止 loading，并展示“继续 / 帮我选一个 / 收敛建议”入口。

### 7.8 final_decision 触发

用户可以通过以下方式触发最终决策：

```text
点击“帮我选一个”
输入“继续”
输入“给最终建议”
输入“收敛建议”
```

前端沿用当前 `convergenceDeckForMessage` / `convergeProductDeck` 逻辑即可，但需要保证当前 deck 已经是最新 deck。

## 8. 目标体验

用户输入：

```text
推荐适合油皮的洗面奶，200元以内
```

理想体验：

```text
系统：正在检索匹配商品
→ 候选商品卡组出现，展示 2-5 个商品
→ 筛选调整卡出现，告诉用户当前按“油皮 / 洁面 / 200 元以内 / 日常护肤”筛选
→ 用户可以直接滑商品，也可以点筛选卡调整条件
→ 用户也可以直接说“不要酒精”“再温和一点”
→ 用户说“继续”或点“帮我选一个”
→ 系统生成 final_decision
```

## 9. 验收标准

### 9.1 P0 Demo 验收

输入：

```text
推荐适合油皮的洗面奶，200元以内
```

必须满足：

```text
出现 ProductRecommendationStrip
ProductRecommendationStrip 内至少 1 个商品，建议 2-5 个
商品卡可点击进入 Swipe / 详情
商品来自库内数据
不编造价格、优惠、库存
done.finish_reason = awaiting_product_feedback
done.deck_id 非空
```

### 9.2 结构化筛选验收

同一轮或紧随商品卡之后必须出现 `criteria_card`，用于解释当前筛选范围。

`criteria_card` 必须支持：

```text
展示关键筛选字段
打开编辑底板
发送 criteria_patch
重跑推荐并生成新 deck
提示用户也可以自然语言调整
```

自然语言调整也必须支持：

```text
“再温和一点” → 新推荐更偏温和
“不要酒精” → 新推荐排除含酒精商品或给出无匹配说明
“预算再低点” → 新推荐降低预算上限
```

### 9.3 澄清验收

输入：

```text
推荐一款商品
```

可以出现 `clarification`，询问品类或关键入口信息。

但输入：

```text
推荐适合油皮的洗面奶，200元以内
```

不应停在澄清或标准确认，应直接进入商品结果。

### 9.4 收敛验收

用户对 deck 做反馈后输入：

```text
继续
```

系统应：

```text
读取当前 deck_id
读取反馈
输出 final_decision
不重新生成一组无关 product_card
```

## 10. 最终取舍

我们不是取消 `criteria_card`，而是改变它的位置和职责。

最终策略：

```text
clarification：模糊时先问
product_card：默认先给结果
criteria_card：后置筛选调整
final_decision：反馈后收敛
```

比赛默认路径：

```text
用户输入 → product_card 自动出现 → 满足官方 P0
```

产品增强路径：

```text
product_card → criteria_card / 自然语言调整筛选 → swipe 反馈 → final_decision
```

这样既满足评委“说完需求就能看到推荐商品”的要求，也保留 BuyPilot 的核心创新：结构化筛选卡、SwipeDeck 反馈、最终决策收敛。
