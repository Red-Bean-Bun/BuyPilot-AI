# 商品候选收敛流程前后端协作说明

## 背景

本文档用于对齐 Android 前端与后端 Agent 在商品推荐阶段的交互节奏。它只描述前端侧的 UI 与交互设计，以及前端期望后端配合支持的接口契约；本次不要求前端修改后端代码，也不直接替换现有后端 PRD 主流程。

当前后端推荐阶段会自动从 `product_card` 继续输出到 `final_decision`。这对“快速给出答案”是顺畅的，但会压缩用户查看候选商品、表达偏好、再让 AI 收敛建议的空间。前端希望将商品推荐阶段拆成两个明确动作：先给候选，再基于用户反馈收敛最终建议。

## 前端设计思路

商品候选区不是完整商城列表，也不是必须完成的任务流。它是 AI 导购在给最终建议前，向用户展示候选并收集偏好信号的轻量交互层。

前端期望流程如下：

```text
用户提出需求
  -> criteria_card
  -> ProductRecommendationStrip 缩略候选
  -> 用户可选查看、喜欢、排除、看证据
  -> 点击「收敛建议」
  -> final_decision
```

聊天流内只展示 `ProductRecommendationStrip` 缩略候选带。每个缩略卡展示商品图、品牌、名称、价格和 1 到 2 个标签。用户点击缩略卡后进入独立挑选或详情体验，可以左滑不喜欢、右滑感兴趣、查看大图详情、下滑看更多信息，并从右上角入口查看推荐证据。

候选商品不应成为强制步骤。用户不需要点满几个商品才能继续，可以直接点击「收敛建议」。如果用户已经查看、喜欢或排除了部分商品，这些行为会作为偏好信号，让最终建议更贴合用户。

前端在候选区展示轻状态，例如：

```text
已看 2 个｜感兴趣 1 个｜排除 1 个
```

候选区主按钮为：

```text
收敛建议
```

可选次级动作包括：

```text
再换一批
补充要求
```

## 当前冲突

现有后端自动输出：

```text
criteria_card -> product_card* -> text_delta -> final_decision -> done
```

这会导致前端刚展示候选商品，最终建议就已经到达。用户还没有机会通过预览、滑动、查看详情或查看证据来表达偏好，`final_decision` 就已经完成，削弱了 SwipeDeck 或缩略候选带在推荐系统中的价值。

前端期望商品候选阶段先停在：

```text
criteria_card -> product_card* -> done
```

其中 `done.payload.finish_reason` 建议为：

```text
awaiting_product_feedback
```

收到该状态后，前端关闭 loading，并在候选区展示「收敛建议」按钮。此时用户可以直接收敛，也可以先通过商品预览、喜欢、不喜欢、看证据等动作提供反馈。

## 接口协作建议

### 1. 保留现有反馈接口

继续使用现有 `POST /feedback` 记录用户行为。建议后端至少识别以下动作：

| action / feedback_type | 含义 | 用途 |
|---|---|---|
| `view_detail` | 用户打开商品详情 | 表示用户对该商品产生兴趣，但不等于喜欢 |
| `like` | 用户右滑或点击喜欢 | 正向偏好，用于最终决策或重排 |
| `not_interested` | 用户左滑或排除 | 负向偏好，用于排除或降低权重 |
| `open_evidence` | 用户查看推荐证据 | 表示用户在关注可信依据 |

示例：

```json
{
  "session_id": "sess_xxx",
  "product_id": "p1",
  "feedback_type": "like",
  "action": "like",
  "reason": "用户右滑感兴趣"
}
```

### 2. 新增收敛建议接口

建议后端新增：

```text
POST /recommendation/converge
```

用途：基于当前 `session_id`、`deck_id` 和已记录的反馈，生成最终购买建议。该接口不再返回新的 `product_card`，只负责给出收敛后的解释与最终决策。

请求体建议：

```json
{
  "session_id": "sess_xxx",
  "deck_id": "deck_xxx",
  "client_turn_id": "turn_client_xxx",
  "selected_product_ids": ["p1"],
  "liked_product_ids": ["p1"],
  "disliked_product_ids": ["p2"],
  "viewed_product_ids": ["p1", "p2"],
  "feedback_summary": "用户看过 p1、p2，喜欢 p1，排除 p2"
}
```

字段说明：

| 字段 | 必填 | 说明 |
|---|---|---|
| `session_id` | 是 | 当前会话 ID |
| `deck_id` | 是 | 当前候选商品组 ID |
| `client_turn_id` | 否 | 前端请求轮次 ID，用于关联 UI 状态 |
| `selected_product_ids` | 否 | 用户显式选择或停留较久的商品 |
| `liked_product_ids` | 否 | 用户喜欢的商品 |
| `disliked_product_ids` | 否 | 用户排除的商品 |
| `viewed_product_ids` | 否 | 用户打开详情或证据的商品 |
| `feedback_summary` | 否 | 前端汇总的人类可读反馈，便于后端 prompt 或日志使用 |

返回方式建议继续使用 SSE 流式响应，复用现有事件类型：

```text
thinking
text_delta
final_decision
done
```

示例返回序列：

```text
event: thinking
data: {"event":"thinking","display_mode":"inline_thinking","payload":{"phase":"decision_generation","message":"正在结合你的选择收敛建议..."}}

event: text_delta
data: {"event":"text_delta","display_mode":"inline_text","payload":{"message_id":"msg_converge_001","delta":"我会优先考虑你感兴趣的商品，同时排除你不喜欢的候选。","done":false}}

event: final_decision
data: {"event":"final_decision","display_mode":"summary_card", "...":"..."}

event: done
data: {"event":"done","display_mode":"none","payload":{"finish_reason":"completed"}}
```

### 3. 首轮候选阶段的完成状态

候选阶段完成时，后端建议返回：

```json
{
  "event": "done",
  "deck_id": "deck_xxx",
  "display_mode": "none",
  "payload": {
    "deck_id": "deck_xxx",
    "total_products": 3,
    "finish_reason": "awaiting_product_feedback"
  }
}
```

前端收到该状态后的行为：

- 停止 thinking 和底部 loading。
- 保留 `ProductRecommendationStrip`。
- 展示「收敛建议」主按钮。
- 允许用户继续输入文字补充要求。
- 允许用户直接点击「收敛建议」，即使没有任何商品反馈。

## 前端兜底策略

如果后端暂未支持 `POST /recommendation/converge`，前端可以先保留 UI 入口，并采用临时兜底策略：

- mock 模式中演示候选后收敛体验。
- fallback 模式中将「收敛建议」转成一次普通 `/chat/stream` 文本请求，例如“基于我刚才看的候选商品，收敛一个最终建议”。
- 若后端仍自动返回 `final_decision`，前端可将它弱化展示为“初步建议”，并保留“基于我的选择再收敛”入口。

这些兜底只用于前端演示和兼容，不代表最终协议。

## 商品详情接口边界

当前 Android SwipeCard 全流程只消费真实后端 `product_card` 已有字段：

```text
product / reason / risk_notes / evidence / actions
```

商品详情页和证据页不会编造 SKU、FAQ 或用户评价。它们只展示商品基础信息、推荐理由、风险提示、匹配标签与证据片段。如果后续需要展示多规格 SKU、完整 FAQ、用户评价或更长的商品资料，建议后端新增 `GET /products/{product_id}` 详情接口，或在不破坏现有 SSE 契约的前提下扩展 `ProductPayload`。

## 验收标准

- 后端同学能明确理解：`product_card` 阶段不是最终答案，前端需要一个候选后暂停的用户控制点。
- 前端同学能明确实现：缩略候选带、可选反馈、轻状态统计和「收敛建议」主 CTA。
- 协议职责清晰：用户行为先通过 `/feedback` 沉淀，最终建议通过 `/recommendation/converge` 触发。
- 不新增 SSE 事件类型，收敛阶段复用 `thinking`、`text_delta`、`final_decision`、`done`。
- 不强制用户点满商品。没有反馈也可以直接收敛，有反馈则建议更精确。
