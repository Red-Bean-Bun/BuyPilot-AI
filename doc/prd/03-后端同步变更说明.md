# 后端同步变更说明

本文档用于对齐后端、Agent、Android 三方的 A2UI 协议口径。目标不是压缩后端工作量，而是把事件流做成可工程化落地的状态机接口。

## 这次改了什么

1. `thinking` 不再是顶部状态条文案，而是当前 AI 回复节点的一部分。
2. `criteria_card`、`product_card`、`final_decision` 都改为”对话区摘要 + 底板详情”的分层呈现。
3. `product_card` 不再是一张张独立长卡，而是同一轮的 `ProductSwipeDeck` item。
4. 证据不直接塞进聊天流，只通过 `evidence_refs` 绑定到商品详情和最终依据底板。`evidence_refs` 增加 `snippet` 字段直接携带证据摘要文本。
5. 停止生成改为底部输入区动作，前端关闭 SSE 连接为主要取消信号，同时 best-effort 调用 `POST /chat/cancel`。后端同时支持连接断开检测和显式 cancel 两种方式。
6. `session_id` 首次创建由后端负责：首次请求 session_id 为 null 时后端生成，在所有事件 envelope 中返回。前端从首个事件提取并缓存。
7. 反馈回流机制明确为：后端通过 `session_id` 查询 feedbacks 表获取历史反馈，注入 criteria 约束。不依赖 history 字段。
8. `done` 事件 payload 增加 `client_turn_id` 回传字段，供前端关联请求与响应。
9. `final_decision` 的 `next_actions` P0 阶段不包含 `compare`（P1 功能）。
10. 删除 `POST /stream/token` 端点（Stream SDK 已砍）。

## 后端需要实现的协议能力

### 统一 envelope

所有 SSE 事件都要带：
- `schema_version`
- `session_id`
- `turn_id`
- `seq`
- `event_id`
- `node_id`
- `deck_id`
- `display_mode`
- `created_at_ms`
- `payload`

### 事件语义

- `thinking`：inline thinking 节点，支持阶段文案更新和心跳。
- `clarification`：完整澄清卡，可直接回复。
- `criteria_card`：摘要卡，完整字段与编辑项进入底板。
- `text_delta`：同一 `node_id` 追加增量文本。
- `product_card`：同一 `deck_id` 下的 swipe deck item。
- `final_decision`：摘要卡，查看依据进入底板。
- `done`：标记当前轮结束。
- `error`：错误节点，保留在聊天流内。

## 新增

1. `POST /chat/cancel`，按 `session_id + turn_id` 取消当前轮。后端同时支持 SSE 连接断开检测。
2. 后端在 `criteria_card`、`product_card`、`final_decision` 中返回可直接映射底板的数据分层。
3. 后端保证 `seq` 单调、`event_id` 幂等、`node_id` 稳定。
4. `evidence_refs` 增加 `snippet` 字段，直接携带证据摘要文本，前端无需额外 API 调用。
5. 首次请求 `session_id` 为 null 时，后端生成 `sess_{uuid}` 格式的 session_id。
6. `done` 事件 payload 增加 `client_turn_id` 回传。
7. 反馈注入通过 `session_id` 查询 feedbacks 表实现，不依赖 history 字段。
8. 澄清回复流程：后端通过 session_id 查询上一轮 partial_criteria 自动恢复上下文。
9. 删除 `POST /stream/token`（Stream SDK 已砍）。

## Android 侧会怎么用

- 用 `node_id` 做 upsert。
- 用 `deck_id` 聚合商品卡。
- 用 `display_mode` 决定摘要、详情、底板还是 deck。
- 用 `evidence_refs` 打开证据底板，不在聊天流直接展示长证据。
