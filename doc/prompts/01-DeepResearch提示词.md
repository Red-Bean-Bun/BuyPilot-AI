## 为什么拆分

原 prompt 同时要求模型扮演：调研员、架构师、产品经理、iOS工程师、后端工程师、RAG工程师、数据工程师、评测工程师、项目经理、答辩教练。这不是 DeepResearch，这是让一个虚拟顾问模拟一家创业公司，还不发工资。

拆分后每轮只解决一个决策面，最后再收束。产出链：

```
边界收缩 → 模块调研 → 技术选型 → 架构整合 → API/Schema 契约 → 三周执行 → 答辩包装
```

## 使用顺序

**第一批并行**（可以一起跑）：
1. 总策略与 MVP 边界
2. Android 用户端
3. FastAPI 后端
4. RAG / 向量数据库
5. Rerank / 多模态
6. 轻量后台
7. 数据集
8. 评测框架
9. 开源风险

**第二批整合**：10. 最终技术选型与架构整合

**第三批落地**：11. 代码级契约

**第四批执行**：12. 三周计划与答辩包装

如果时间不够，最低限度跑这 5 个：1、2、4、8、10。

## 公共上下文（每次跑 DeepResearch 时先贴这段）

```
我们正在参加 ByteDance AI Full-Stack Challenge / AI 全栈挑战赛。

课题：Agent 智能体——实现一个基于 RAG 的多模态电商智能导购 Agent。

目标：3 人组队，3 周内完成高完成度、可演示、可答辩、可评测的比赛作品。

团队约束：
1. 3 人团队，开发周期 3 周，使用 AI Coding / vibe coding
2. 客户端：Android 原生（Kotlin + Jetpack Compose + Stream Chat Android Compose SDK）
3. 后端：Python FastAPI + PostgreSQL + pgvector
4. LLM：百炼平台（Qwen-Turbo / Qwen-Plus / Qwen-VL-Plus / text-embedding-v3 / gte-rerank）
5. 系统三部分：用户端 Android App、轻量 Web 后台、后端服务

产品战略：
1. 品类聚焦亲子玩具（Toys & Games），前台可包装泛电商，后台做深玩具数据
2. 不做普通聊天机器人，核心是"购物决策 Agent"
3. 用户缺的不是更多信息，而是判断标准
4. 混合检索：SQL/JSON filter（适龄/预算/安全）+ vector retrieval（语义）+ rerank
5. 评测闭环必须作为答辩亮点

MVP 保留：文字输入、图片输入、流式对话、商品卡片、左滑不喜欢、右滑喜欢、点击详情、最终决策卡
MVP 暂缓：上滑/下滑复杂手势、复杂抽屉、长期用户画像、完整商家后台、复杂多 Agent 编排

研究要求：输出中文，每个关键结论给来源链接，站在 3 人 3 周比赛交付负责人角度做取舍。
```

## 每轮 DeepResearch 后的压缩模板

每次 DeepResearch 输出通常很长。在每个提示词末尾追加这段，让它附一个"可转交摘要"：

```
最后请额外输出一个"供下一轮研究使用的压缩摘要"，不超过 800 字，包含：
1. 本轮最终推荐
2. 不推荐方案
3. 关键证据链接
4. 最大风险
5. 后续研究需要继承的约束
```

## 最关键的补充提示（防止温吞废话）

在所有 DeepResearch prompt 最后追加：

```
请不要给出"方案 A/B/C 都可以"的模糊结论。你必须站在比赛交付负责人角度做取舍，明确指出：最应该选什么、坚决不选什么、为什么不选。请把"3 周内能稳定交付"放在第一优先级，把"技术看起来高级"放在第二优先级之后。任何会显著增加集成风险但不能显著提升答辩说服力的技术，都应该被降级或砍掉。
```
