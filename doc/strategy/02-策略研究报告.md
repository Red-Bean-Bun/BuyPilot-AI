# 基于 RAG 的多模态电商智能导购 Agent：字节跳动 AI 全栈挑战赛战略与 MVP 边界专项研究报告

## 执行摘要

在当前大模型技术栈快速迭代与智能体（Agent）概念全面落地的背景下，构建基于检索增强生成（RAG）的多模态电商智能导购 Agent，已经成为验证人工智能商业价值与全栈工程落地能力的高潜力场景。针对"三人、三周"的极端开发周期与字节跳动 AI 全栈挑战赛的特定约束条件，本研究报告旨在提供一套端到端的战略规划、架构取舍逻辑与最小可赢产品（Minimum Winnable Product, MWP）的严格边界定义。

研究分析表明，在 AI 辅助编程（Vibe Coding / AI Coding）高度普及的当下，全栈开发赛事的考察核心已经发生了根本性的转移。**传统的底层代码编写能力已退居次席，取而代之的是产品定义精度、架构级开源组件（Glue Code）的集成效率、端到端用户体验的完整性，以及系统自我迭代能力的构建**。

### 最终技术决策

| 决策项 | 选择 |
|--------|------|
| 品类 | **多品类**（美妆护肤/数码电子/服饰运动/食品生活），导师提供官方数据 |
| 客户端 | **Android 原生**：Kotlin + Jetpack Compose + OkHttp SSE + 自研 ChatUiNode 卡片流 |
| LLM 提供商 | **双轨并行**：火山引擎 Doubao（意图识别主力）+ 百炼 Qwen（生成主力），按任务类型选 primary/fallback |
| 后端 | Python FastAPI + PostgreSQL + pgvector + SQLModel |
| 流式协议 | SSE（Server-Sent Events），Android 端通过 OkHttp SSE 直连 FastAPI `/chat/stream` |
| 模型切换 | 配置层 dict + call_llm() 函数，按任务类型选 primary/fallback |

> **模型-任务映射**：意图识别→Doubao(primary)/Qwen-Turbo(fallback)；购买标准生成→Qwen-Plus(primary)/Doubao(fallback)；推荐解释生成→Qwen-Plus(primary)/Doubao(fallback)；多模态图片→Qwen-VL-Plus(primary)；Embedding→text-embedding-v3(百炼)；Rerank→gte-rerank(百炼)。

为了在三周内最大化胜率，团队必须实施极为苛刻的范围控制（Scope Control）。通过详尽的 P0/P1/P2 优先级划分，战略性地剥离非核心的电商交易链路（订单、支付、库存管理、商家入驻等），将极其有限的研发资源压倒性地投入到以下五个维度：

1. **购买标准生成机制**（模糊需求 → 结构化决策标准）
2. **多模态意图精准解析**（文字 + 图片）——加分项4.2 ⭐⭐⭐拍照找货
3. **混合检索防幻觉机制**（硬过滤 + 向量召回 + Rerank + 证据绑定）
4. **Android 原生流式体验**（媲美豆包的流式交互 + Agent-to-UI 卡片渲染）
5. **对话式加购能力**（⭐入门加分项，展示 Agent 对结构化数据的 CRUD 操作）

> **评审权重对齐**：基础功能完整性35% + 工程质量25% + 效果与可靠性20% + 加分项深度20%。P0/P1/P2 划分逻辑从"时间驱动"改为"评审权重驱动"——P0 服务35%+25%底线，P1 服务20%+20%提升，P2 只做打磨。

最终战略目标不是交付一个大而全但体验粗糙的泛电商聊天机器人，而是交付一个高度聚焦、体验极度流畅、具备自驱动评测反馈闭环的 **多品类智能导购决策 Agent**。

## 赛题本质判断

### 2.1 工程完整度与端到端交付的决定性地位

在业界高规格的全栈黑客松中，评审委员会往往将项目的整体完成度与业务链路闭环置于首位考量，而非某一个单一算法模型的微小精度提升。一个包含致命漏洞、极易崩溃的"多智能体协同系统"，在现场演示时的得分，将远远落后于一个架构简单但运行极度稳定、用户界面丝滑且逻辑严密的"单智能体混合检索系统"。

端到端交付的本质要求是消除链路中的任何断点：

> Android 端输入文本/上传图片 → FastAPI 接收 → 意图识别 → 约束抽取 → 购买标准生成 → 混合检索（结构化过滤 + 向量召回 + Rerank） → LLM 生成带证据的解释 → FastAPI SSE 事件流 → Android OkHttp SSE 接收 → ViewModel/StateFlow → 自研 LazyColumn + ChatUiNode 协同渲染 → 用户反馈记录 → 评测闭环更新

这种工程完整度直接映射了团队对复杂系统集成、异步状态管理、移动端主线程调度、网络异常处理、流式 UI 渲染和后端服务稳定性的把控能力，是区分成熟开发者与初学者的核心标尺。

### 2.2 AI 辅助编程（Vibe Coding）的驾驭能力

现代 AI 开发赛事的入场券是熟练运用 Cursor、GitHub Copilot、Claude Code 等工具进行自动化代码生成。评审委员会默认参赛队伍不再需要花费数天时间去手写基础 CRUD、常规 UI 页面或模板化接口代码。

团队的核心竞争力便转化为：

- **客户端**：Jetpack Compose 自研 LazyColumn + ChatUiNode 抽象负责聊天 UI（消息列表、输入框、图片附件、卡片流）；OkHttp SSE 负责 RAG 流式输出直连 FastAPI
- **网络层**：Retrofit（普通业务）+ OkHttp SSE（RAG 流式输出直连 FastAPI）
- **本地数据**：Room（缓存会话/卡片/反馈）+ DataStore（轻量配置）+ ViewModel/StateFlow（统一状态管理）
- **图片采集**：Android Photo Picker（轻量 MVP）/ CameraX（后续扩展）+ ML Kit（端侧 OCR 预处理，可选）

过度执着于手写底层实现不仅会消耗宝贵比赛时间，更会被视为未能有效利用现代 AI 生产力工具的战略失误。

### 2.3 RAG 能力的深度考察：从被动检索到主动推理

传统 RAG 往往停留在简单的"向量检索 + 文本总结"层面，但在电商导购场景中存在严重局限。评审的核心关注点不在于系统能否把文档塞进向量数据库，而在于系统能否处理复杂、模糊、多约束的用户查询。

例如：
> "给4岁孩子买个室内益智玩具，预算200以内，不要小零件"

这需要先提炼决策标准（适龄 3-6 岁、室内场景、益智维度、安全约束），再基于标准检索，而非简单关键词匹配。

系统必须采用：
- **SQL/JSON Filter**：处理价格、适龄、安全特征等刚性约束
- **Vector Retrieval**：处理"益智"、"室内安静"等抽象语义
- **Cross-Encoder Rerank**：综合硬条件 + 软偏好 + 证据重排
- **LLM Recommendation Explanation**：带证据绑定的可解释推荐

RAG 能力的本质不是"能搜到东西"，而是"能基于可靠证据减少幻觉并辅助决策"。

### 2.4 Android 原生客户端体验与多模态能力的融合

赛题强调客户端需要利用原生技术栈打造接近"豆包"的流式交互体验。Android 端的主技术路线：

**Kotlin + Jetpack Compose + OkHttp SSE + ViewModel/StateFlow + Room + Coil + Photo Picker**

三者分工清晰：
- **OkHttp SSE**：解决 RAG 流式传输（thinking → criteria_card → product_card → cart_action → final_decision）
- **Compose LazyColumn + ChatUiNode**：统一承载文本消息与结构化卡片（购买标准卡、商品卡、证据卡、风险提醒卡、最终决策卡）

Android 客户端必须严控主线程阻塞。流式渲染、Markdown 解析、图片压缩都不能压在 UI Thread 上。使用 Kotlin Coroutines 将网络请求和图片处理放到 IO Dispatcher，UI 层只消费 ViewModel 暴露的状态流。

**Android 客户端不是一个"壳"，而是作品体验的第一战场。** 评委看到的不是数据库和向量索引，而是用户输入后 App 是否流畅响应、是否能逐步生成购买标准、是否能漂亮地呈现商品卡片、是否能把用户反馈实时影响下一轮推荐。

### 2.5 质量评测与反馈闭环（Eval Loop）的工业级标准

建立稳健的质量评测与反馈闭环，是区分"玩具级演示项目"与"工业级原型系统"的核心分水岭。

Android 客户端承担关键数据入口：左滑不喜欢、右滑喜欢、点击详情、停留时长、最终采纳等行为，都应被记录为反馈事件并写入后端 feedback 表。

轻量 Web 后台负责展示评测闭环：不同 Prompt 版本的得分变化、加入结构化过滤前后的准确率变化、加入 Rerank 前后的召回质量变化、典型失败案例、用户负反馈高频原因等。

别人只是"能聊"，我们是"能评估、能解释、能迭代"。

## 产品战略评估

### 3.1 为什么以导师官方数据驱动多品类导购：回归课题原意

导师提供的官方数据覆盖美妆护肤、数码电子、服饰运动、食品生活 4 个品类（每类25条），与课题说明会列举的典型场景（"油皮洗面奶"、"蓝牙耳机"、"推荐跑鞋"）完美匹配。在官方数据基础上构建多品类导购能力，是回归课题原意的务实选择。

美妆护肤品类天然适合展示反选排除能力——"不要含酒精的"比玩具的"不要电池款"更贴近真实消费者决策场景，且反选排除需依赖语义理解而非纯SQL字段过滤，更能体现 RAG + LLM 的协同能力。

答辩叙事顺序：

1. 为什么用导师数据做多品类？——课题场景完美匹配，数据质量碾压自构造数据
2. 每条商品有 marketing_description + FAQ + reviews，天然适合 RAG chunking
3. 反选排除在美妆品类更真实——"不要含酒精的"需要语义理解，不是SQL字段过滤
4. 我们用4品类验证了"一套可迁移的导购决策架构"，反选排除靠语义而非结构化字段

### 3.2 "决策辅助"优先于"商品搜索"：高度合理

将智能体的核心定位锚定在"决策辅助"而非单纯"商品搜索"，是该产品的灵魂所在。

**消费者购物的本质，是不断降低资金投入与预期收益之间的不确定性。** 当用户提出"给4岁孩子买个室内益智玩具，不要小零件"时，传统关键词搜索往往失效。真正的智能导购应主动把模糊需求转化为判断标准，再基于这些标准进行检索与推荐。

### 3.3 "购买标准生成"作为核心模块：高度合理

在面对参数繁杂的玩具选择时，家长最大的痛点在于"不知道该看什么"。系统若仅罗列商品，反而会加剧选择困难。

该模块机制：
> 用户输入："给4岁孩子买个室内益智玩具，预算200以内，不要小零件"
> → 系统生成结构化标准：age 3-6、indoor、budget ≤ 200、safety: no_small_parts、education: logic/fine_motor
> → Android 端以购买标准卡显性展示
> → 用户确认/调整后 → 混合检索 → 推荐生成

本质上是把**专家思维可视化**。

### 3.4 混合检索优先于纯向量检索：高度合理

在电商导购场景中依赖纯向量检索将导致灾难性后果。向量检索擅长捕捉语义相似度，但对适龄范围、价格上限、安全特征等刚性条件并不天然可靠。

系统必须利用 SQL/JSON Metadata Filter 处理硬性约束，利用向量检索处理抽象语义，最后通过 Rerank 重排。

### 3.5 轻量后台只做知识库和评测闭环：高度合理

将管理后台的范围严格限定在知识库维护与评测闭环可视化，完美贴合三人三周的苛刻资源约束。

轻量后台职责限定为：
- 商品结构化数据维护
- 商品详情/营销文档上传
- 知识库切分与向量入库状态查看
- 用户反馈记录查看
- 评测任务触发与结果可视化
- Prompt/RAG 策略版本对比

明确不做订单、库存、支付、商家入驻、多角色审核等完整商家后台能力。

## 最小可赢 MVP（Minimum Winnable Product）

### 4.1 最小可运行版本（Minimum Runnable Version）

**客户端架构**：基于 Kotlin + Jetpack Compose 构建基础导购对话界面，用 LazyColumn + ChatUiNode 抽象承载消息列表与卡片流。P0 完成 OkHttp SSE 接入 FastAPI `/chat/stream`，实现最小流式链路。

**后端服务**：搭建 Python FastAPI 服务，提供 `/chat/stream` SSE 流式接口。LLM 接入双轨策略——Doubao（火山引擎Ark）用于意图识别，Qwen-Plus（百炼）用于生成/决策，配置层 dict + call_llm() 实现按任务类型切换 + fallback。

**检索与数据引擎**：导师提供的**100条**脱敏电商数据（4品类×25，中文，含 marketing_description + FAQ + reviews + SKUs + 本地图片）直接灌入 PostgreSQL + pgvector，实现最基础的混合检索（结构化过滤 + 向量召回）。rag_knowledge 天然 chunking，无需额外切分策略。

**核心目标**：第 7 天前跑通主链路——文本输入 → FastAPI RAG → 购买标准生成(初版) → 推荐生成(初版) → OkHttp SSE 流式返回 → LazyColumn + ChatUiNode 渲染商品卡片。验收标准是**评委拿起App能走通完整链路**，不是"代码写完了"。

### 4.2 最小可演示版本（Minimum Demoable Version）

**客户端**：在 LazyColumn + ChatUiNode 基础上，完成 Agent-to-UI 结构化流式渲染。接收 `thinking`、`clarification`、`criteria_card`、`text_delta`、`product_card`、`cart_action`、`final_decision`、`done` 等 SSE 事件。购买标准卡、商品推荐卡、证据卡、最终决策卡通过 Compose 自定义组件动态插入消息流。

**后端**：构建完整 RAG 管道，实现意图识别、约束抽取、购买标准生成、混合检索（硬过滤 + pgvector + Rerank）、推荐生成。文档自动切分、Embedding、向量化入库。多模态接入百炼 Qwen-VL-Plus。

**评估与后台**：建立基础 Web 管理页面，支持查看商品数据、文档入库状态、用户反馈和评测数据。

**核心目标**：系统能完美应对预设演示路径：用户输入需求或上传图片 → 生成购买标准 → 混合检索 → 返回推荐卡片 → 后台对应日志和评测数据可见。

### 4.3 最小可赢版本（Minimum Winnable Version）

这是本次比赛真正的交付目标：

- **极致 Android 流式体验**：自研 LazyColumn 聊天容器 + FastAPI SSE 低延迟流式事件 + Compose 自定义 Agent 卡片，组合成接近 ChatGPT/豆包的导购体验
- **A2UI 式结构化渲染**：后端返回结构化 JSON（thinking 状态、购买标准、候选商品、推荐理由、证据来源、风险提醒、最终决策），Android 端动态渲染不同卡片
- **高级认知与决策链路**：双轨模型架构——Doubao-Seed-2.0-lite 负责意图识别/路由（primary），Qwen-Plus 负责购买标准生成/推荐解释/决策卡（primary），各自有 fallback
- **工业级混合检索与防幻觉机制**：文本向量检索 + 商品属性结构化过滤 + gte-rerank 重排 + 证据绑定。任何推荐都绑定证据来源
- **对话式加购能力**：⭐入门加分项——用户说"把这个加到购物车"，Agent识别意图并执行加购，客户端展示购物车状态。证明Agent能通过对话操作结构化数据

## 四条 Demo 场景路径设计（3.5分钟，对齐官方评审场景）

### 场景 1：模糊推荐与条件筛选（1分钟）

**用户输入**："推荐适合4岁孩子的益智玩具"

**系统内部动作**：
1. **意图识别**（Doubao primary）→ recommend
2. **购买标准生成**（Qwen-Plus primary）→ age 3-6、education: logic/fine_motor
3. **混合检索**：SQL过滤超龄商品 → 向量检索"益智"语义匹配 → 推荐生成
4. 流式输出：thinking → criteria_card → product_card → cart_action → final_decision → done

**评委感知点**：完整端到端链路跑通 + 结构化购买标准。

### 场景 2：多模态拍照找货（1分钟）

**用户输入**：上传一张玩具包装图 + "这个适合3岁孩子吗？会不会有吞咽风险？"

**系统内部动作**：
1. Qwen-VL-Plus 识别商品类型、品牌、适龄标识
2. RAG 在安全说明文档中检索吞咽风险信息
3. 流式输出："系统识别到这是 XXX 积木玩具，标注适龄 5 岁以上。根据安全说明书，该产品含小颗粒部件，对 3 岁儿童存在吞咽风险。替代推荐：YYY 大颗粒版本，适龄 3+。"

**评委感知点**：⭐⭐⭐多模态完整闭环 + 安全约束硬性执行 + 证据绑定。

### 场景 3：多轮追问与反选排除（1分钟）

**用户输入**：首轮推荐后说"不要电池款" → "预算降到150"

**系统内部动作**：
1. criteria_patch 注入新约束（requires_battery: false, budget_max: 150）
2. 重新混合检索 → 新推荐 + 避坑说明
3. 流式输出：criteria_card(update) → product_card → final_decision → done

**评委感知点**：多轮约束变化 + 反选排除语义理解 + 反馈闭环。

### 场景 4：对话式加购（0.5分钟）

**用户输入**："把这个加到购物车"

**系统内部动作**：
1. 意图识别 → add_to_cart
2. 写入 cart_items 表
3. SSE 返回 cart_action 事件（action: add, product_id, status: success）
4. 客户端购物车图标数字变化 + 商品卡片"已加购"标记

**评委感知点**：Agent 通过对话操作结构化数据（CRUD）——从"推荐咨询"升级到"交易执行"。

## 最大风险与止损策略

### 8.1 风险一：混合检索效果不达标

**现象**：硬性约束（如预算）与模糊语义（如"益智"）结合时，向量检索方向性偏离，推荐出超预算商品。

**止损**：如果 Rerank 调优超过 2 天无进展，降级为保守但安全的"前置硬性漏斗过滤"策略——LLM 将自然语言转 SQL/JSON 查询，先在结构化数据库框定硬性条件子集，再在封闭子集中做向量语义计算。在玩具导购领域，宁可推荐略显平庸，也绝不能突破年龄安全红线。

### 8.2 风险二：Android 流式卡片渲染卡顿

**现象**：SSE 高频数据 + Compose 动态卡片插入 + 流式文本动画导致主线程阻塞，UI 掉帧。

**止损**：P0 即接入 OkHttp SSE + LazyColumn 最小渲染路径，不堆复杂动画。如果复杂结构化卡片在高频流式中无法平滑解析，降级为：后端先流式输出纯文本稳住体验，文本流完成后发送独立附加组件数据包，前端瞬间渲染静态商品卡。牺牲少量动态感，换取全局不崩溃。

### 8.3 风险三：双轨模型切换不稳定

**现象**：Doubao API 调用失败（限流/超时/格式不兼容）时 fallback 切换不顺畅，链路中断。

**止损**：call_llm() 函数内置 retry 逻辑——primary 失败后自动切换 fallback，第二次失败则降级为最保守输出（预设模板回复而非 LLM 生成）。绝不让一个模型故障导致整条链路崩掉。

### 8.4 风险四：答辩时无法解释代码原理（官方减分项）

**现象**：代码由AI大量生成，答辩时评委问"为什么这样设计"无法脱口而出。

**止损**：建立 design-decisions.md，每个核心决策3句话：选了什么/为什么/反过来会怎样。答辩前自读一遍。

## 最终建议与定位

在一个仅有 3 人的微型团队中，面对 3 周倒计时的巨大压力，制胜之道不是盲目堆砌冗余代码，而是：

1. **建立极其冷酷的系统边界控制**
2. **将 AI 辅助编码生产力压榨到极限**
3. **精准狙击评审委员会隐性评估痛点**

系统在任何交互触点上，都必须表现为一个充满底气、逻辑严密、拥有高智商的"多品类智能导购专家"。所有隐藏在冰山之下的混合检索、防幻觉机制、评测反馈闭环，以及展现在水面之上的 Android 原生级流式渲染，都必须高度统一并服务于一个最纯粹的商业目标：

**摧毁并消除消费者的决策迷茫。**

### 最终一句话破局定位

**"我们并不是在做一个大而全的泛电商客服机器人，而是利用混合 RAG 架构，构建了一个能够将用户模糊的购物需求精准转化为可量化的购买标准，通过多品类数据和多模态解析实现强可解释性推荐，以反选排除验证语义理解防幻觉能力，并自带端到端质量评测进化闭环的多品类智能导购决策智能体。"**

死守这一战略边界，冷酷果断地砍掉一切分散注意力的外围功能。确保第 7 天雷打不动让评委拿起App能走通完整链路（文字输入→意图→购买标准→检索→推荐→卡片渲染），第 14 天4条Demo路径全部可演示且无幻觉，并在最后一周穷尽一切力量把 Android 端的流式 UI、商品卡片、购买标准卡和 Demo 剧本打磨到近乎产品级的流畅。
