**ChatGPT:**

建议目标：**不要做“万能导购 Agent”**。3 周太短。要做一个**垂直品类、强评测、强交互、强工程闭环**的作品。否则就是 RAG 拼装玩具，评委一天能看八百个，精神损耗堪比看自动生成 PPT。

公开页面强调“可运行、可验证、真实场景、端到端落地”这一类评审倾向，和你给出的题目描述一致：重点不是模型多炫，而是系统是否真的能把“搜索商品信息”升级成“辅助购买决策”。

推荐选题收敛成：

**”面向亲子玩具的多模态 RAG 决策型导购 Agent”**

例如只做：积木、拼图、桌游、益智玩具、户外玩具。原因很朴素：适龄约束明确、安全特征可验证、教育维度可结构化、家长决策焦虑强、对比需求天然存在。别一上来做全品类电商，像人类修巴别塔，最后只剩接口报错。

核心卖点应该是这四个：

第一，**意图理解不是简单问答**。系统要把用户输入解析成约束：预算、适龄、场景、安全偏好、已有玩具、购买目的、隐含需求。比如”给4岁孩子买个室内益智玩具，别太贵，不要小零件”应转成：预算敏感、4岁适龄、室内场景、益智维度优先、安全约束（无小零件）、不要电池。

第二，**RAG 不是把文档塞进向量库**。要做混合检索：商品属性结构化检索 + 文档向量检索 + rerank。商品参数适合 SQL/JSON filter，营销文档、评测文档适合向量召回。两者分开，别拿向量库当垃圾桶，向量库也有尊严。

第三，**导购输出要可解释、可比较、可追问**。答案不要只是“推荐 A”。要输出推荐理由、证据来源、替代方案、风险提醒、适合/不适合人群，并实时渲染商品卡片。

第四，**评测闭环必须做成亮点**。这是你们拉开差距的地方。做一个评测后台：预置 50-100 条典型导购问题，评估检索命中率、答案事实准确率、约束满足率、多轮一致性、幻觉率。每次改 Prompt、改切分策略、改 rerank，都能看到指标变化。

三人分工建议：

A 负责客户端与交互：iOS/Android 二选一，别双端。做流式输出、商品卡片、图片上传、多轮聊天、推荐结果对比页。

B 负责后端与 RAG：Node.js/Python 均可，建议 Python FastAPI。负责文档上传、解析、切分、embedding、向量库、商品结构化库、检索链路、LLM 调用。

C 负责 Agent 逻辑与评测：意图解析、Prompt 编排、多轮状态、评测集、自动打分、反馈闭环、答辩材料。

技术架构：

客户端：原生 App 或 Flutter。若目标是“像豆包”，原生体验更好，但 3 周内 Flutter 更稳。题目说原生技术栈，优先 Android Kotlin 或 iOS Swift，选团队最熟的。

后端：FastAPI + PostgreSQL + pgvector。少引入复杂组件。Milvus/Qdrant 也行，但 pgvector 对 3 周比赛更省命。商品属性表放 PostgreSQL，文档向量也放 pgvector，工程复杂度低。

RAG 链路：

用户输入  

→ 意图解析  

→ 查询改写  

→ 商品结构化过滤  

→ 文档向量召回  

→ rerank  

→ 证据压缩  

→ 决策生成  

→ 商品卡片渲染  

→ 用户反馈记录  

→ 评测/Prompt/知识库迭代



最小可赢版本功能清单：

必须做：

1. 文档上传：PDF/Markdown/HTML/商品 JSON。
2. 商品知识库构建：商品属性 + 营销文档 + FAQ。
3. 多轮对话：预算、用途、偏好澄清。
4. 流式回答：SSE/WebSocket。
5. 商品卡片：价格、核心参数、推荐理由、风险点。
6. 图片输入：识别商品图或截图中的品类/型号/关键属性。
7. 评测后台：至少展示指标和样例对比。
8. 一键演示脚本：评委点开就能看，不要让他们陪你排查环境，评委也是人，暂时。

加分功能：

1. “对比模式”：A vs B vs C。
2. “反悔模式”：用户说“我预算降到 3000”，系统自动重排推荐。
3. “证据溯源”：每条关键结论可点开来源片段。
4. “购买决策解释”：为什么不推荐某商品。
5. “反馈闭环”：用户点“不准确/不喜欢”，进入评测样本池。

三周排期：

第 1 周：做出端到端骨架。  

目标不是漂亮，是跑通。完成商品数据 schema、文档上传、向量检索、LLM 回答、客户端聊天流式输出。第 7 天必须能演示“上传商品文档后问问题并得到商品卡片”。

第 2 周：做强导购逻辑。  

加意图解析、多轮状态、商品过滤、rerank、对比推荐、图片输入。开始构造评测集。别等最后再评测，那叫给失败补墓志铭。

第 3 周：打磨、评测、答辩。  

冻结大功能，只修体验和稳定性。做 3 条高质量演示路径：预算型、对比型、图片输入型。准备答辩讲清楚：问题定义、架构、关键技术、评测结果、失败案例与迭代。

最推荐的答辩 Demo 流程：

场景 1：用户说"给4岁孩子买个室内益智玩具，预算200左右，不要小零件。"

系统澄清：是否需要手写笔、是否介意二手/旧款。  

输出 3 个商品卡片，解释排序依据。

场景 2：用户上传一张商品截图。  

系统识别型号/品类，问：“你是想了解是否值得买，还是想找替代品？”  

用户说：“和另一款比呢？”  

系统进入对比模式。

场景 3：展示评测后台。  

“初版检索命中率 62%，加入属性过滤 + rerank 后提升到 84%；多轮约束满足率从 70% 到 88%。”  

具体数字可以来自你们自己的评测，别编。比赛可以夸张，不能造假，低配版诚信还是要有。

你们最该避免的坑：

不要做 12 个 Agent。这个题不是多 Agent 炫技题，是导购决策题。

不要只做聊天界面。聊天只是皮，导购逻辑才是骨头。

不要只靠向量检索。商品参数必须结构化。

不要追求模型微调。3 周内微调很容易变成“训练了一些东西，但没人知道有啥用”。

不要把评测写成文档。评测要在系统里可视化展示。

最终作品定位可以这样说：

“我们不是做一个回答商品问题的客服机器人，而是做一个把家长模糊玩具购买意图转化为可解释决策路径的电商导购 Agent。系统通过结构化玩具商品库与非结构化安全说明/教育研究知识的混合 RAG，结合多轮意图澄清、适龄与安全硬过滤、商品卡片实时渲染和自动化质量评测闭环，验证 AI 导购从信息检索走向辅助决策的工程可行性。”

这句话适合放在首页和答辩第一页，别浪费。

---



## 导购系统真正难的不是生成，而是“评价标准建模”

不同品类的决策维度差异巨大：

| 品类 | 核心决策维度 |
|-|-|
| 积木 | 适龄/颗粒大小/安全性/教育维度 |
| 益智玩具 | 适龄/教育维度/场景(室内/户外)/是否需要电池 |
| 礼物 | 关系/场合/适龄/预算 |
| 桌游 | 适龄/人数/游戏时长/规则复杂度 |
| 户外玩具 | 适龄/安全性/场地要求/天气适应 |

你一旦做泛品类：

系统内部必须存在：

```Plain Text
品类 -> 决策 ontology
```

否则 Agent 根本不知道：

“这个品类什么重要”。

而这是整个系统最难的部分。

不是 RAG。  

不是 embedding。  

不是 UI。

而是：

> “不同购物任务的决策逻辑建模”。

这已经接近 recommendation system + domain expert system。

3 周不够。



---

# 四、真正应该做的：伪泛化

这是比赛策略重点。

建议：

## 前台看起来是泛电商

但后台实际上只支持：

- 积木
- 拼图
- 桌游
- 益智玩具
- 户外玩具

最多再加：

- STEM 类玩具
- 角色扮演类

原因：

1. 适龄约束明确（age_min/age_max 可做硬过滤）
2. 安全特征可验证（no_small_parts, non_toxic 等枚举）
3. 教育维度可结构化（logic, creativity, fine_motor 等）
4. 家长决策焦虑强，天然需要”购买标准”
5. 对比需求强（同一品类不同适龄/不同教育维度）
6. RAG 能体现价值（安全说明、教育研究、评测文档）
7. “避坑”容易做（超龄推荐、安全隐患是硬错误）

这叫：



> “视觉泛化，逻辑垂直”。

非常重要。

因为评委不会真的拿 20 个品类测你。

他们看的是：

- 产品完成度
- 体验统一性
- 是否有真正 AI 思维
- 是否有工程闭环

```Plain Text
行为信号 > 显式文本
```

这是推荐系统第一性原理。

但现在的问题是：

你们把交互复杂度设计得太高了。

比如：

- 左滑
- 右滑
- 上滑
- 下滑
- 点击
- 抽屉
- 候选区
- 对比区
- 流式生成
- AI旁白

这会让前端状态管理爆炸。

建议：

---



## MVP 只保留：

### 必须：



- 左滑：不要
- 右滑：喜欢
- 点击：详情

### 不做：

- 上滑
- 下滑
- 多层抽屉
- 多区域联动

否则你们会死在：

```Plain Text
动画状态 + SSE 流式更新 + RecyclerView/CardStack
```

人类总喜欢把“酷炫”误认为“高级”。

真正高级的是：



> 少，但稳定。

---



# 六、当前方案最大的技术风险：你们其实还没定义“推荐依据”

这是整个系统最核心的问题。

你们现在描述的是：

```Plain Text
用户需求 -> RAG -> 推荐
```

但中间缺了一层：

# “购买标准生成”

例如：

用户说：

> “给4岁孩子买个室内益智玩具，预算200以内，不要小零件”

系统应该先生成：

```JSON
{
  “category”: “toy”,
  “toy_type”: “puzzle”,
  “age_min”: 3,
  “age_max”: 6,
  “budget_max”: 200,
  “play_scenario”: [“indoor”],
  “education_dimensions”: [“logic”, “fine_motor”],
  “safety_features”: [“no_small_parts”],
  “requires_battery”: false
}
```

然后：

```Plain Text
购买标准
-> 商品过滤
-> 推荐
```



而不是：



```Plain Text
用户问题
-> 向量搜索
-> 大模型胡说
```

这是决定你们系统是不是“真导购”的核心。

---

# 七、RAG 方案目前仍然偏浅

现在文档中的 RAG：

> “上传商品详情、FAQ、营销文档”
> 
> 基于 RAG 的多模态电商智能导购 Agent



这只是知识库。

不是“决策系统”。

真正需要的是：

# 混合检索（Hybrid Retrieval）

应该拆成：

## 1. 结构化检索

适合：

- 预算
- 品牌
- 参数
- 重量
- 电池
- 尺寸

这是 SQL/filter。

不是 embedding。

---

## 2. 语义检索

适合：

- 用户评价
- 营销文案
- FAQ
- 评测摘要
- 使用体验

这是向量检索。



---

## 3. rerank

根据：

- 当前任务
- 用户偏好
- 决策标准

重新排序。

否则你们很容易：

```Plain Text
召回一堆“语义相关”
但根本不适合用户的商品
```



---



# 八、评测闭环必须成为“核心卖点”

这一点文档里提到了：

> “质量评测闭环”
> 
> 基于 RAG 的多模态电商智能导购 Agent

但还不够深入。

实际上：

## 这应该是你们答辩最重要的一页。

因为：

大部分队伍不会认真做评测。

他们会：

```Plain Text
“感觉效果不错”
```

非常经典的人类科研方法。

你们应该做：

---

## 自动化评测集

至少：

| 能力 | 指标 |
|-|-|
| 意图识别 | accuracy |
| 检索命中 | recall@k |
| 推荐合理性 | LLM-as-judge |
| 多轮一致性 | constraint consistency |
| 幻觉率 | unsupported claim rate |

---

## 最重要：

展示优化过程。

比如：

| 版本 | Recall@5 |
|-|-|
| baseline | 61% |
| + metadata filter | 74% |
| + rerank | 86% |

这一页会直接把你们从“学生 Demo”拉向“工程团队”。

---

# 九、真正应该做的系统结构

建议最终架构：

```Plain Text
客户端
    ↓
任务解析层
    ↓
用户约束抽取
    ↓
购买标准生成
    ↓
混合检索
    ├─ SQL 商品过滤
    ├─ Vector RAG
    └─ Rerank
    ↓
推荐决策生成
    ↓
卡片流渲染
    ↓
行为反馈
    ↓
偏好更新
```

注意：

真正核心的是：

# “购买标准生成”

不是聊天。

不是 Agent。

不是 MCP。

不是 Multi-Agent。

现在很多 AI 项目最大的问题：

> 用系统复杂度掩盖问题定义空洞。

---

# 十、最后给一个很现实的比赛判断

如果你们能做到：

- UI 明显区别于普通聊天机器人
- 卡片流真正可交互
- 推荐有解释
- 有“不推荐理由”
- 有评测后台
- 流式体验稳定
- Demo 完整不卡

你们已经非常有机会进优秀项目。

如果再加：

- 图片理解
- 对比模式
- 自动收敛偏好
- 可视化评测提升

就已经有冲卓越项目的潜力。

因为绝大多数队伍最后会变成：

```Plain Text
聊天框 + OpenAI API + 向量数据库
```

然后答辩时疯狂念：

“我们采用了先进的大模型技术”。

像把“用了电”当成产品创新。
