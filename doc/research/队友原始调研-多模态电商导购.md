# 一、文档目的与当前交接边界

这份文档不是最终技术架构文档，也不细化数据库表、接口字段或具体模型选型。它的目的，是把当前产品负责人的设计思考、业务判断、前端交互方向和 MVP 边界先交接给团队，用于确认 idea 是否成立。

后续如果团队认可该方向，再由后端 / Agent 架构同学继续设计整体架构、数据流、接口边界、RAG 入库与检索策略，以及质量评测闭环。

**先确认“我们要做一个什么样的产品”，再讨论“系统应该怎么实现”。**

# 二、赛题理解：不要做成普通 AI 对话应用

赛题要求实现一个基于 RAG 的多模态电商智能导购 Agent，打通“意图理解 - 智能咨询 - 决策辅助”的核心路径，并在客户端提供接近豆包式的流式交互体验。

这里的关键并不是简单复刻一个豆包聊天窗口，而是要让用户在购物决策过程中感到系统实时、连续、可解释、能响应反馈。

因此，我们的设计不应停留在“聊天框 + AI 回复 + 商品卡片”的常规形态，而应把 AI 对话能力和电商决策过程重新组织成一种更轻、更快、更有产品辨识度的体验。

**AI 对话框负责解释与陪伴，卡片流负责决策与筛选。**

# 三、第一性原理思考

## 3.1 用户购物的本质不是搜索商品，而是降低决策不确定性

用户进入电商平台时，并不总是缺少商品。恰恰相反，很多时候用户面对的是过多商品、过多参数、过多评价和过多营销信息。真正的问题不是“找不到商品”，而是“不知道哪个商品适合我”。

所以，这个 Agent 的底层命题不应是“让 AI 帮用户搜索商品”，而应是“让 AI 帮用户减少购买不确定性”。

- 用户不是想要更多商品，而是想要更确定的选择。
- 用户不是缺少信息，而是缺少判断标准。
- 用户不是只想知道商品有什么，而是想知道商品是否适合自己。
- 用户害怕买错、买贵、买回来不适合，所以需要风险提醒和不推荐理由。

## 3.2 泛电商 Agent 应该面向所有品类，但不能让用户承担复杂度

这个 Agent 不应被限制为某一个单一品类。用户可能买数码、服饰、美妆、家居、礼物、食品或运动用品。不同品类的判断标准不同，但购物决策的底层结构是相通的。

- 先理解用户想完成什么购物任务。
- 再识别商品品类、预算、场景、偏好和禁忌。
- 然后建立适合本次任务的购买标准。
- 最后通过商品 / 方案对比、风险提醒和最终建议帮助用户收敛决策。

因此，产品前台要保持轻量，后台可以根据不同任务和品类调用不同导购策略。

## 3.3 前台极简，后台完整

系统内部可以有完整的流程：意图识别、品类判断、场景约束提取、RAG 检索、商品召回、推荐解释、反馈记录和质量评测。

但用户前台不应该感知到复杂流程。用户只需要做三件事：说出需求、刷卡筛选、得到结论。

**系统复杂度应该由 Agent 承担，而不是由用户承担。**

# 四、业务目标

本项目的业务目标，是构建一个面向泛电商场景的 AI 购物决策 Agent，帮助用户从“我想买”走到“我知道为什么买”。

- 降低用户购物时的信息理解成本和选择成本。
- 通过任务入口和卡片流交互，让用户更快表达偏好。
- 通过 RAG 知识库让推荐理由有依据，增强用户信任。
- 通过刷卡反馈和对话补充，让系统动态理解用户偏好。
- 通过最终决策卡给出首选、备选、不推荐原因和购买前提醒。

# 五、用户问题

从用户视角看，电商购物中的问题可以被概括为以下几类：

| **用户问题** | **具体表现** | **产品应对方式** |
|-|-|-|
| 需求模糊 | 用户只说“推荐一个耳机”“想买礼物”，但没有完整条件 | 通过任务类型和轻量问题卡澄清关键条件 |
| 信息过载 | 商品太多，详情页、参数、评价太杂 | 用卡片化结果和简短理由降低阅读负担 |
| 判断标准缺失 | 用户不知道本次购物应该优先看哪些指标 | 在后台建立购买标准，并在前台用标签或简短解释表达 |
| 害怕买错 | 担心商品不适合、超预算、踩雷 | 提供避坑提醒、不推荐原因和购买前提醒 |
| 表达偏好困难 | 用户很难用长文本准确描述喜好 | 通过刷卡行为收集偏好，让系统越刷越懂用户 |

# 六、产品方案：任务入口 + 流式对话 + 决策卡片流

## 6.1 产品定位

本项目不定位为普通 AI 客服，也不定位为传统商品搜索工具，而是定位为：

**一个面向泛电商场景的刷卡式 AI 购物决策 Agent。**

用户通过文字或图片创建购物任务，系统在后台完成意图理解、RAG 检索和推荐生成，在前台以豆包式流式对话和 Tinder 式刷卡卡片流的组合方式呈现。

## 6.2 类型选择作为轻量快捷入口

类型选择快速帮助系统判断用户是在“选商品”“做方案”“比较商品”“根据图片推荐”“避坑”还是“挑礼物”。

但类型选择不应成为强制门槛。用户可以点任务类型，也可以直接输入需求，由系统自动识别任务。

| **购物任务类型** | **用户本质需求** | **前端入口表达** |
|-|-|-|
| 帮我选商品 | 我知道大概买什么，但不知道选哪个 | 推荐一个适合我的商品 |
| 帮我做方案 | 我不是买单品，而是想解决一个场景问题 | 生成一套商品组合方案 |
| 帮我比较 | 我已有候选，需要判断哪个更适合 | 对比几个商品的取舍 |
| 根据图片推荐 | 我有图片场景、风格或商品，希望 AI 理解 | 上传图片并生成推荐 |
| 帮我避坑 | 我想知道某个商品值不值得买 | 分析风险和不推荐原因 |
| 帮我挑礼物 | 我不知道送什么，怕不合适 | 根据关系、预算、场合推荐 |

## 6.3 豆包式流式交互的理解

赛题要求“在客户端提供接近豆包式的流式交互体验”。这里不应被理解为只能做普通聊天 UI。更合理的理解是：系统要在用户输入后即时响应、逐步生成、连续反馈。

- 文本流式输出：AI 对话框逐步解释当前理解和推荐思路。
- 状态流式反馈：任务摘要、识别结果和推荐状态逐步出现。
- 卡片流式生成：商品卡、方案卡、避坑卡不是一次性列表，而是像发牌一样逐步生成。
- 行为流式响应：用户每次刷卡后，AI 对话框即时总结偏好变化。

**像豆包一样即时响应，像 Tinder 一样轻松筛选，像购物参谋一样给出最终建议。**

# 七、核心业务流程

为了减少用户心智负担，用户前台感知到的流程应尽量压缩。复杂能力在后台完成，前台保持“输入 - 刷卡 - 调整 - 总结”的轻量路径。

1. 用户进入 App，可以直接输入购物需求，也可以选择购物任务类型：帮我选商品、帮我做方案、帮我比较、根据图片推荐、帮我避坑、帮我挑礼物。
2. 系统根据用户输入或任务类型，流式识别用户意图、商品品类、预算、场景和偏好，并在界面顶部生成轻量任务摘要。
3. 如果关键信息不足，系统只插入一张关键问题卡，让用户通过点击快速补充，避免连续问答带来的心智负担。
4. 系统在 AI 对话框中流式说明推荐思路，同时逐步生成商品卡、方案卡、避坑卡或对比卡，形成刷卡式决策流。
5. 用户通过左右滑、上下滑、点击详情等方式表达喜欢、不适合、想了解或暂时跳过。
6. AI 对话框实时解释当前推荐、总结用户偏好变化，并允许用户补充“预算提高一点”“不要这个品牌”“更适合送人”等自然语言。
7. 系统根据刷卡行为和对话补充动态调整后续推荐，让推荐结果逐步收敛。
8. 当用户留下足够候选后，系统流式生成最终决策卡，给出首选、备选、不推荐原因和购买前提醒。
9. 用户可以查看商品详情、收藏、继续调整或提交反馈，反馈进入后续评测与知识库优化。

**一句话版本：输入需求，开始刷卡，AI 越刷越懂你，最后给出购买结论。**

# 八、前端体验设计

## 8.1 页面结构

客户端页面不应以聊天框为唯一中心，而应形成“任务状态 + 卡片流 + AI 对话框”的三层结构。

- 顶部：轻量任务摘要，展示当前任务、预算、场景、偏好等关键信息。
- 中间：刷卡式决策卡片流，承载商品卡、方案卡、避坑卡、对比卡等核心内容。
- 底部或右侧：AI 对话框，用于流式解释、偏好总结和自然语言补充。

## 8.2 卡片类型分层

为了避免流程过重，卡片应分为“主流程卡片”和“按需辅助卡片”。

| **层级** | **卡片类型** | **使用方式** |
|-|-|-|
| 主流程卡片 | 商品卡、方案卡、关键问题卡、最终决策卡 | 用户必须或高频看到，承担核心决策流程 |
| 辅助卡片 | 需求画像、购买标准、避坑卡、对比卡、证据卡 | 按需展示、穿插出现或通过抽屉展开，不强制每次出现 |

需求画像和购买标准不一定要作为完整页面出现，可以被压缩到顶部任务摘要、商品卡标签或 AI 对话解释中。

## 8.3 滑动操作含义

刷卡不是为了好玩，而是为了降低用户表达偏好的成本。用户不必用长文本描述偏好，而是通过行为让系统逐渐理解。

| **操作** | **用户含义** | **前端表达** |
|-|-|-|
| 右滑 | 感兴趣 / 加入候选 | 卡片进入候选区，AI 提示已记录偏好 |
| 左滑 | 不适合 / 排除 | 卡片移出，系统降低相似特征权重 |
| 上滑 | 想了解更多 | 展开详情、证据或推荐依据 |
| 下滑 | 暂时跳过 | 不强烈影响推荐，继续下一张 |
| 点击 | 查看详情 / 对比 / 收藏 | 进入详情层或候选对比 |

## 8.4 AI 对话框的角色

AI 对话框保留，但它不是主流程，而是“购物参谋旁白 + 用户补充入口”。它应该尽量短句化、状态化、解释化，而不是连续输出长篇文本。

- 解释当前为什么推荐这张卡。
- 总结用户最近的刷卡偏好。
- 提示系统接下来如何调整推荐。
- 接收用户自然语言补充，例如“预算提高一点”“不要这个品牌”“更适合送人”。

# 九、技术方案方向

本节只描述技术方向，不展开具体数据结构和接口字段。

根据赛题要求，系统需要围绕 iOS / Android 原生客户端、Node.js / Python / Go 后端服务、向量数据库和大模型 OpenAPI 形成完整链路。

- 客户端：面向 iOS / Android 原生框架设计，重点实现流式对话、卡片流实时渲染、图片上传、刷卡交互和反馈提交。
- 后端服务：可采用 Node.js / Python / Go 等技术栈，负责任务管理、会话调度、商品数据服务、Agent 调度和反馈记录。
- RAG 知识库：通过上传非结构化商品详情、营销文档、FAQ 和评价摘要构建专属知识库。
- 向量数据库：用于存储商品文档向量，支持基于用户需求的语义检索。
- 大模型 OpenAPI：负责意图理解、轻量追问、推荐解释、多模态图片理解和最终决策总结。
- 质量评测：围绕典型导购场景评估回答准确率、知识检索精度、多轮对话逻辑和用户反馈。

# 十一、开发联调与 MVP 边界

## 11.1 MVP 优先跑通的主链路

1. 用户进入 App，直接输入需求。
2. 系统返回轻量任务摘要和第一组卡片。
3. 用户通过刷卡表达偏好。
4. AI 对话框根据刷卡行为进行短句反馈。
5. 系统生成后续推荐卡片。
6. 用户留下候选后，系统生成最终决策卡。
7. 用户提交反馈，后台记录用于评测。

## 11.2 暂不优先做的内容

- 真实电商平台全量数据接入。
- 复杂用户画像和长期个性化推荐。
- 完整商家后台和复杂运营权限体系。
- 所有品类的深度规则；第一版先验证泛流程，再扩展品类策略。
