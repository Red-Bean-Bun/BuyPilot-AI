# 数据库设计 Review（2026-05-31）

## 审查范围

- 数据库文档：`doc/database/`
- 仓储实现：`backend/src/repos/`
- 相关数据链路：`backend/src/services/product_ingest.py`、`backend/src/services/retriever.py`、`backend/src/services/eval/`
- 需求基准：`doc/official/课题说明会-基于RAG的多模态电商智能导购AI-Agent.md`

## 总体结论

当前数据库设计能支撑比赛 MVP：商品事实层、RAG chunk、会话状态、反馈、购物车、trace、证据绑定、评测闭环和审计日志都已覆盖。整体方向符合课题对“RAG 链路可靠性”“不编造商品事实”“多轮上下文”“反选排除”“对话式加购”和“可解释推荐”的要求。

主要风险不在表数量，而在三个边界：

1. 结构化硬过滤没有下推到数据库，pgvector Top-K 之后才过滤，规模扩大或 query 偏移时可能漏掉正确商品。
2. 商品事实源存在 `data/raw` 和 `products` 表双轨，数据库本身还不能单独证明商品卡片事实与 chunk 索引一致。
3. schema 迁移、约束和并发安全偏 demo 化，适合当前 100 条数据，但不够像可长期演进的数据库设计。

## 原始 Findings

以下问题描述保留审查时的原始表述；当前修复状态以上一节“2026-05-31 修复落地状态”为准。

### 1. High：检索硬过滤没有下推到数据库

**位置**

- `backend/src/repos/documents.py:72`
- `backend/src/services/retriever.py:134`
- `doc/official/课题说明会-基于RAG的多模态电商智能导购AI-Agent.md:47`
- `doc/official/课题说明会-基于RAG的多模态电商智能导购AI-Agent.md:68`

**问题**

`list_vector_chunks_by_similarity()` 的 pgvector SQL 只过滤：

- `embedding IS NOT NULL`
- `metadata.retrieval_role != risk`

品类、预算、商品类型、品牌排除、成分排除等“硬约束”在召回后的 Python 层 `_passes_hard_filters()` 执行。这意味着数据库先按全库向量相似度取 Top-K，再从 Top-K 里过滤。当前只有 100 条商品、1292 个 chunk，问题不明显；但数据量稍大或 query 语义偏移时，符合预算/品类的商品可能在 Top-K 外，被提前截断。

**影响**

官方基础场景要求“结构化参数提取 + 范围过滤”，数据治理要求价格等关键参数准确。若过滤发生在向量 Top-K 之后，系统可能出现：

- 明明库里有符合预算/品类的商品，却返回空或预算放宽结果。
- trace 看起来有向量召回，但不容易解释“为什么正确商品没进候选”。
- 后续多品类扩展时 pgvector 召回成本和误召回概率上升。

**建议**

- `product_chunks` 与 `products` join 查询，把 `category`、`price`、`brand`、`sub_category/product_type` 等可结构化条件下推到 SQL。
- PostgreSQL 上补索引：
  - `products(category, price)`
  - `products(category, sub_category)`
  - `products(brand)`
  - `product_chunks(product_id)`
  - 如继续依赖 JSONB metadata，再补必要 GIN/表达式索引。
- trace 中保留 `pre_filter_count`、`post_filter_count`、`sql_filters_applied`，方便答辩解释过滤损耗。

### 2. High：商品事实源存在 raw JSON 与 products 表双轨

**位置**

- `backend/src/repos/products.py:1`
- `backend/src/services/product_ingest.py:43`
- `backend/src/services/retriever.py:134`
- `doc/database/tables/products.md:5`

**问题**

代码注释明确 raw JSON 是 single source of truth。运行时商品卡片 payload 由 `backend/src/repos/products.py` 从 `data/raw/ecommerce_agent_dataset` 直接派生；数据库 `products` 表由 seed 过程写入，检索命中 chunk 后再通过 raw cache 的 `get_product()` 组装商品卡。

这造成事实源分叉：

- `products` 表文档称其是商品事实源。
- 运行时商品事实实际上来自 raw JSON cache。
- `product_chunks` 来自某次 seed 的 raw 派生产物。

如果 raw JSON 变更但未 reindex，商品卡片和 chunk 证据可能来自不同版本。

**影响**

官方减分项强调不能编造商品、价格、优惠信息。当前设计能防止 LLM 编造，但数据库层不能单独证明“商品卡片价格/图片/品牌”和“召回 chunk 证据”来自同一数据版本。

**建议**

二选一：

1. 以数据库为运行时事实源：商品卡片、硬过滤和购物车全部读 `products` 表；raw JSON 只作为 seed 输入。
2. 保留 raw 为事实源，但把 `products/product_chunks` 明确定义为派生索引表，并增加：
   - `dataset_version`
   - `source_file`
   - `source_hash`
   - `embedded_at`
   - `embedding_model`
   - `chunking_version`

启动或 reindex 后校验 raw、products、chunks 的版本一致。

### 3. Medium：schema 迁移机制不足，文档与运行时数据库策略不一致

**位置**

- `backend/src/repos/database.py:56`
- `backend/src/repos/database.py:173`
- `backend/src/config/settings.py:96`
- `doc/database/README.md:3`

**问题**

当前建表逻辑使用 `SQLModel.metadata.create_all()`，再补缺失列。这只能处理新增列，不能可靠处理：

- 列类型变更，例如 JSON -> JSONB、JSON embedding -> vector。
- 索引新增/删除。
- 唯一约束、检查约束、外键策略变更。
- 字段重命名或语义迁移。

同时，数据库文档仍写“当前开发库 SQLite；生产目标 PostgreSQL + pgvector”，但 `settings.py` 已经禁止普通运行时 SQLite，仅允许 pytest 下 `/tmp` SQLite。

**影响**

评委或队友审计时会看到两个叙述：

- 文档：当前库是 SQLite dev.db。
- 代码：运行时必须 PostgreSQL + pgvector。

这会削弱数据库设计可信度，也让后续迁移风险难以追踪。

**建议**

- 在 `doc/database/README.md` 明确：运行时目标和 demo 验收均为 PostgreSQL + pgvector；SQLite 只用于 pytest 隔离。
- 增加轻量迁移清单或 Alembic。
- 至少建立 `schema_version` 表，记录当前 schema、chunking、embedding 版本。
- 不要让 `create_db_and_tables()` 承担破坏性迁移，继续保持显式 reindex/drop 脚本。

### 4. Medium：购物车缺少并发安全的唯一约束和原子 upsert

**位置**

- `backend/src/repos/models.py:70`
- `backend/src/repos/cart_items.py:23`
- `doc/database/tables/cart_items.md:30`
- `doc/official/课题说明会-基于RAG的多模态电商智能导购AI-Agent.md:151`

**问题**

`cart_items` 当前通过代码层先查 `session_id + product_id`，存在则递增数量，不存在则插入。表上没有唯一约束。

当前 demo 单用户低并发通常没问题，但两个并发加购请求可能同时查不到记录，最终插入两条同商品购物车行。

**影响**

购物车 CRUD 是课题加分项。并发重复行会导致：

- 购物车展示重复商品。
- 总价和数量计算偏差。
- 后续“删除第二个”“把数量改成 2”等自然语言操作语义不稳定。

**建议**

- 增加唯一约束：`UNIQUE(session_id, product_id)`。
- PostgreSQL 使用原子 upsert：
  - `INSERT ... ON CONFLICT (session_id, product_id) DO UPDATE SET quantity = cart_items.quantity + EXCLUDED.quantity`
- 如果未来做 SKU 级购物车，唯一键改为 `(session_id, product_id, sku_id)`，并补 `price_snapshot`。

### 5. Medium：评测 ground_truth 是自由 JSON，仍混用旧约束命名

**位置**

- `backend/src/types/sse_events.py:62`
- `backend/src/repos/models.py:145`
- `backend/src/services/eval/metrics.py:59`
- `data/eval/eval_samples.json`

**问题**

运行时 `Constraints` 已经是封闭 DSL，字段包括：

- `budget_max`
- `product_type`
- `use_scenario`
- `ingredient_avoid`
- `brand_avoid`
- `origin_avoid`

但评测样本和 deterministic metrics 仍使用部分旧命名：

- `max_price`
- `min_price`
- `must_have_features`
- `forbidden_features`
- `use_case`
- `brands`

`eval_samples.ground_truth` 本身是自由 JSON，没有 schema version，也没有 seed 时校验。

**影响**

评测闭环可能低估或误判真实系统能力。例如运行时正确抽取 `budget_max=200`，但 ground truth 用 `max_price=200`，指标需要靠别名逻辑才能对齐；一旦别名覆盖不完整，评测结果就不可信。

**建议**

- 给 `ground_truth` 增加 `schema_version`。
- seed 时校验 ground truth key 必须属于当前 `Constraints` 或明确的 eval-only 字段。
- 将历史 key 迁移到当前 DSL：
  - `max_price -> budget_max`
  - `min_price -> budget_min`
  - `use_case -> use_scenario`
  - `forbidden_features -> ingredient_avoid` 或 eval-only `forbidden_terms`
  - `must_have_features -> product_type/ingredient_prefer/dietary` 或 eval-only `must_match_terms`
- 指标代码中保留兼容层，但报告里标明迁移状态。

## 正向评价

### 商品事实层与 RAG chunk 分层合理

`products` 保存商品基础事实，`product_chunks` 保存可检索证据，符合 RAG 系统职责拆分。chunk metadata 中包含 `chunk_type`、`retrieval_role`、`evidence_kind`，能解释召回来源，也能避免风险/差评 chunk 被当成正向卖点召回。

相关位置：

- `backend/src/repos/models.py:19`
- `backend/src/repos/models.py:33`
- `backend/src/services/chunking.py:138`
- `doc/database/tables/product_chunks.md:5`

### trace 与 evidence_links 是可解释推荐核心资产

`retrieval_traces` 保存过滤、向量召回、rerank、最终选择和阶段耗时；`evidence_links` 保存商品与证据 chunk 的绑定。这直接服务“为什么推荐这些商品”和“防幻觉”。

相关位置：

- `backend/src/repos/models.py:113`
- `backend/src/repos/models.py:128`
- `backend/src/repos/traces.py:19`
- `doc/database/tables/retrieval_traces.md:5`
- `doc/database/tables/evidence_links.md:5`

### 观测与审计分表正确

`api_request_logs` 记录 HTTP 请求事实，`audit_events` 记录业务副作用。这个拆分比把所有日志塞进一张表更容易审计和回放。

相关位置：

- `backend/src/repos/models.py:156`
- `backend/src/repos/models.py:175`
- `doc/database/tables/api_request_logs.md:5`
- `doc/database/tables/audit_events.md:5`

### 当前裁剪符合比赛周期

没有做订单、支付、库存、商家后台、长期用户画像，符合项目裁剪。对于三周 hackathon，当前 schema 更像“可演示 RAG 导购闭环”，不是完整电商交易系统，这是合理取舍。
