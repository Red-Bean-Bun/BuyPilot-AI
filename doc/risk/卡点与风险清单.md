# 可能的卡点

会卡，而且会卡得很有层次。你们这个项目不是”写一个 App”，而是同时碰了 **Android 端、轻量后台、后端服务、RAG、结构化检索、多模态、评测闭环、答辩包装**。

你的细化方案已经把系统拆成用户端、后台端、后端服务三部分，并且明确 Android Kotlin/Jetpack Compose、轻量 Web 后台、FastAPI/RAG/评测接口这些边界，这个判断是对的，但也意味着卡点会从“技术不会”升级成“系统集成地狱”。

我提前按严重程度预判：

## 最大卡点：范围失控

最危险的不是某个库不会用，而是你们每个人都觉得“这个功能很小”。然后第 12 天系统里出现：图片识别、卡片滑动、RAG、rerank、后台上传、评测看板、多轮对话、最终决策卡、Prompt 版本对比。每个都半成品，合起来就是一座现代软件废墟。

止损规则要提前写死：

第 7 天没有端到端 Demo，就砍掉所有 P2。

第 14 天没有稳定主链路，就砍掉多模态增强和复杂后台。

第 18 天之后不许加新功能，只修 bug、做评测、打磨答辩。

比赛里功能数量不值钱，稳定演示值钱。

## Android 端会卡在”流式对话 + 卡片流状态管理”

Jetpack Compose 做普通页面很快，但你们要做的是：

用户输入

→ SSE 流式输出

→ 中途插入商品卡片

→ 商品卡片可滑动

→ 用户反馈影响后续推荐

→ 对话和卡片状态同步

这里会产生前端状态机问题。比如 token 还在流，卡片已经出来；用户左滑了一张卡，后端还在生成下一张；网络断了，UI 要不要回滚。人类总喜欢把“像豆包一样流式”说得像泡面一样简单，实际是异步事件编排。

建议：

第一版不要边流式边动态插卡。  
先让后端一次返回 `task_summary + cards + assistant_text`，Android 用假流式动画展示。  
等主链路稳了，再做真正 SSE。

MVP 的卡片交互只保留：

左滑：不喜欢

右滑：喜欢

点击：详情

不要上滑、下滑、抽屉、复杂候选区。那些东西非常适合毁灭周末。

## 后端会卡在“接口契约不稳定”

如果你们前端和后端边写边改 schema，第 5 天开始就会互相折磨。

必须第一天冻结这些 JSON：

`TaskSummary`  
`UserConstraint`  
`BuyingCriteria`  
`ProductCard`  
`RAGEvidence`  
`FinalDecisionCard`  
`FeedbackEvent`  
`StreamEvent`

哪怕字段很粗糙，也要稳定。字段可以为空，不能天天改名。

建议立刻定一个原则：

前端永远只消费卡片 JSON，不关心 RAG 细节。

后端永远输出结构化卡片，不直接让 LLM 生成 UI 文本。

Agent 模块永远只产出“约束、标准、证据、推荐理由”。

这叫边界。没有边界，项目就会变成三个人围着一个 bug 祈雨。

## RAG 会卡在“看起来检索了，实际上没用”

很多 RAG Demo 都是假 RAG：向量库召回几段文本，然后 LLM 继续自由发挥。评委一问“这个推荐依据来自哪里”，现场气温下降三度。

你们要避免这个坑。

真正要做的是：

结构化过滤先筛商品：预算、品牌、类别、价格、参数。

向量检索再找证据：详情、FAQ、营销文档、评价摘要。

rerank 再按用户任务重排。

最后回答必须绑定 evidence。

最小实现可以很朴素：

PostgreSQL 存商品表。  
pgvector 或 Qdrant 存文档 chunk。  
每个 `ProductCard` 带 `evidence_ids`。  
详情页能展示“推荐依据片段”。

不要追求 GraphRAG、复杂 Agent 编排、长期记忆。那是“技术炫耀型自杀”。

## “购买标准生成”会卡在 Prompt 不稳定

这是你们的核心亮点，但也容易变成玄学。

用户说：

“预算 3000，考研用平板，主要看课记笔记。”

系统必须稳定生成：

```JSON
{
  "category": "tablet",
  "budget": {"max": 3000},
  "scenario": ["考研", "看课", "记笔记"],
  "priority": ["手写体验", "续航", "屏幕", "重量"],
  "avoid": ["性能过剩但价格高", "配件生态差"]
}
```

卡点在于：LLM 可能今天输出中文数组，明天输出自然语言，后天少字段。

解决办法：

所有关键 Prompt 强制 JSON schema。

解析失败就重试一次。

第二次失败就降级为默认购买标准。

不要让一个 Prompt 失败导致整条链路崩掉。

工程世界没有“模型应该懂”。只有“字段为空时系统怎么办”。

## 多模态会卡在“图片理解看着炫，但链路价值弱”

图片输入很适合答辩开场，但别把它做成核心依赖。

最稳做法：

图片输入只承担三件事：

识别品类。

识别可能型号或文字。

生成补充检索条件。

不要指望图片模型直接做购买决策。视觉模型识别错商品很常见，尤其是截图压缩、角度、遮挡、中文小字。到时候它说耳机是电饭煲，也不是完全没有可能，人工智能嘛，人类训练出来的。

MVP 里图片链路可以这样：

上传图片  
→ 多模态模型输出 `detected_category / detected_brand / extracted_text / inferred_intent`  
→ 系统展示“我识别到你可能在看 XX，是否继续推荐？”  
→ 用户确认或补充  
→ 进入普通导购链路

这样就算识别不准，也不会崩。

## 评测闭环会卡在“指标听起来高级，但没有 ground truth”

你们说要评估准确率、检索精度、多轮逻辑，这很好。但问题是：

谁是标准答案？

没有标准答案，评测就是摆设。摆设也能答辩，但容易被问穿。

建议你们自己构造 50 条评测样本：

20 条预算/场景型

10 条对比型

10 条避坑型

5 条图片输入型

5 条多轮约束变化型

每条样本至少包含：

用户问题

期望品类

必须满足约束

期望召回商品

期望 evidence 文档

不应推荐商品

评分规则

评测指标先别贪多，P0 保留 4 个：

意图识别准确率

约束抽取准确率

检索 Recall@K

答案证据支持率

答辩时展示：

baseline

- 结构化过滤
- rerank
- Prompt 优化

这比你们硬塞 12 个指标更有说服力。

## 轻量后台会卡在“越做越像管理系统”

你们已经明确后台只服务商品数据、知识库和评测闭环，不做订单、库存、支付、商家入驻，这是正确边界。

但实际开发时很容易膨胀：

“要不要加登录？”

“要不要加商品分类管理？”

“要不要支持批量导入？”

“要不要做权限？”

“要不要做图表筛选？”

不要。

后台 MVP 只有四页就够：

商品列表 / 编辑

文档上传 / 入库状态

反馈记录

评测结果看板

如果时间不够，甚至用 Streamlit 都行。评委不是来采购你们后台 SaaS 的。

## 数据会卡在“没有真实感”

如果商品数据太假，整个导购 Agent 会像玩具。

但如果你们爬真实电商，又会卡在清洗、版权、反爬、字段混乱。人类互联网的馈赠通常附带泥巴。

建议：

使用半模拟数据。

真实世界参数结构 + 自己整理/生成商品文档。

品类聚焦 3C。

商品数量 30-60 个就够。

每个商品至少有：

结构化参数

卖点

缺点

适合人群

不适合人群

FAQ

评价摘要

营销文档

避坑说明

数据质量比数据数量重要。60 个干净商品胜过 6000 条垃圾 CSV。

## 开源胶水会卡在“依赖冲突和不可魔改”

你们准备用开源项目拼装，这是对的。但要记住：

开源项目省的是从 0 到 1 的时间，不省理解成本。

最适合复用：

Compose swipe card / LazyColumn 示例

SSE/WebSocket 库

FastAPI 模板

RAG 框架

向量数据库

后台表格/图表组件

评测框架部分能力

不适合直接套：

购买标准生成

商品卡片 JSON

推荐决策逻辑

反馈导致推荐变化

评测样本设计

答辩 Demo 流程

这些是你们的作品灵魂。直接套开源，评委会闻到那股“GitHub 拼盘”的味道。

## 部署会卡，而且通常在最后一天卡

最后一天部署失败是比赛项目的传统民俗活动。

提前避免：

第 3 天就写 Docker Compose。

第 7 天就在云服务器跑一次。

不要第 20 天才第一次部署。

API key、环境变量、数据库初始化、向量扩展安装，都写进 README。

演示时准备三套方案：

在线版本

本地录屏

本地 mock fallback

不要相信现场网络。现场网络的稳定性通常和人类承诺差不多。

## 团队协作会卡在“大家都在做核心，没人做胶水”

3 人组最容易的问题：

A 做 Android，很漂亮。

B 做后端，很抽象。

C 做 RAG，很聪明。

第 10 天发现三者不能连。

必须设一个“联调优先”规则：

每天晚上合并一次主链路。

所有模块都围绕 Demo 脚本开发。

不要各自闭门造航母。

第 1 天就定：

三条 Demo 路径

全部 API schema

商品卡片结构

反馈事件结构

评测样本结构

## 最终预警清单

你们最可能死在这 5 个点：

1. 功能范围失控。
2. Android 流式 + 卡片状态管理复杂。
3. RAG 没有真正 evidence 绑定。
4. 评测没有 ground truth，变成装饰。
5. 最后部署和联调翻车。

最强止损策略：

第 7 天只追求一条完整链路：

文字输入

→ 任务摘要

→ 购买标准

→ 商品过滤

→ RAG 证据

→ 3 张商品卡

→ 用户右滑/左滑

→ 推荐调整

→ 最终决策卡

→ 评测后台展示一条指标

这条链路跑通，你们就活了。

这条链路没跑通，别谈多模态、别谈 rerank、别谈后台美化、别谈什么 AgentHub 式未来蓝图。

比赛不是证明你们想象力丰富。

比赛是证明你们能在约束下交付。
