# Grill-me 决策记录（2026-05-20）：官方文档对齐迭代

> 来源：grill-me 采访，基于飞书官方课题说明文档与推断文档的差距梳理。
> 共16个决策分支，全部走完。以下为每个决策的完整记录。

## Q1: LLM 提供商

**背景**：官方文档提供了 Doubao-Seed-2.0-lite 的 API Key 和 endpoint（火山引擎 Ark），但用户现有文档全部基于百炼/Qwen。官方同时写了"不限制具体使用模型"。
**选项**：A. 全部迁移到 Doubao / B. 双轨并行（按任务类型选主力，另一个做 fallback）/ C. Qwen为主，Doubao做对比展示
**决策**：**B. 双轨并行**
**理由**：Doubao 是字节自家模型，评委一定会问"为什么不用官方提供的"；但 Doubao 目前生态不一定有 VL 和 Rerank endpoint，百炼在这两点更成熟。生成质量敏感的任务用 Qwen-Plus，成本/政策敏感的任务用 Doubao。

---

## Q2: 模型-任务映射

**背景**：双轨策略确认后，需要为5个 LLM 调用点决定 primary/fallback。
**决策**：

| 任务 | Primary | Fallback | 原因 |
|------|---------|----------|------|
| 意图识别 | Doubao | Qwen-Turbo | Doubao免费额度高，意图识别不需要很强 |
| 购买标准生成 | Qwen-Plus | Doubao | JSON schema强约束输出，Qwen-Plus更稳定 |
| 推荐解释生成 | Qwen-Plus | Doubao | 解释需要高质量 |
| 多模态图片 | Qwen-VL-Plus | 无 | Doubao VL生态不确定 |
| Embedding | text-embedding-v3(百炼) | Doubao-embedding-vision | 百炼有Key+维度确定(1024) |
| Rerank | gte-rerank(百炼) | 无 | Doubao无对应服务 |

**核心逻辑**：生成质量敏感→Qwen-Plus，成本敏感→Doubao，基础设施→百炼。成本不在考虑范围内。

## Q3: 购物车——从"绝对不做"到"可选加分"

**背景**：CLAUDE.md 写"绝对不做：支付与订单系统"，但官方4.1加分项明确把购物车⭐入门列为加分点。加分项权重20%。
**选项**：不做 / 做
**决策**：**做**
**理由**：⭐入门档"对话式加购"只需~2天工作量（1张cart_items表+1个API+1个意图分支+购物车图标），不影响P0节奏。不做则加分项20%只能靠2个方向争，做则多覆盖1个加分赛道。

---

## Q4: 购物车做到哪一档

**背景**：官方有3档（⭐入门/⭐⭐进阶/⭐⭐⭐挑战）。
**选项**：⭐入门 / ⭐⭐进阶 / ⭐⭐⭐挑战
**决策**：**⭐入门档，做到极致**
**理由**：⭐入门已是独立加分点。⭐⭐的NLP操作购物车是意图识别延伸，评审价值跟⭐入门重叠。⭐⭐⭐下单确认需要地址/订单状态机，回到之前正确砍掉的方向。把⭐入门做到体验极致比做到⭐⭐但体验粗糙更有评审冲击力。

---

## Q5: P0/P1/P2 划分逻辑——从时间驱动到评审权重驱动

**背景**：官方评审权重：基础完整性35% + 工程质量25% + 效果可靠性20% + 加分项20%。用户现有P0/P1/P2是"时间驱动"（第7天/14天/18天），没有按评审维度映射。
**选项**：保持时间驱动 / 重新调整为评审权重驱动
**决策**：**重新调整为评审权重驱动**
**理由**：时间驱动的风险——P0可能花大量时间做"工程优雅"（25%），但35%的链路还有断点。评审权重驱动确保P0全部指向35%基础完整性，P1指向效果+加分。

---

## Q6: P0 具体范围

**背景**：P0 定义为"基础完整性35% + 工程底线25%"。
**决策**：P0 必须包含完整的端到端链路——输入→意图识别→购买标准生成(初版)→检索→推荐生成(初版)→SSE→商品卡片渲染。验收标准是**评委拿起App能走通完整链路**，不是"代码写完了"。
**关键变化**：购买标准生成和推荐解释生成的初版必须放进P0，否则链路是断的——用户发文字只回一段无结构废话，没有商品卡片，评委看到的是"聊天机器人"不是"导购Agent"。

---

## Q7: P1 具体范围

**背景**：P1 定义为"效果可靠性20% + 加分项启动20%"。
**决策**：正式版卡片+流式动画、混合检索(硬过滤+向量+Rerank)、多轮上下文+反馈+反选排除、图片上传+VL理解、对话式加购⭐入门、双轨模型+fallback。Web管理后台和评测后台降级到P2最小版。
**理由**：评测后台是工程质量的加分项，可以往后排。P1核心是让4条Demo路径可演示。

---

## Q8: 加分项总策略

**背景**：官方4个加分方向，用户4个方向都伸手但每个都是入门/进阶档。
**选项**：A. 广度策略(4方向各打⭐/⭐⭐) / B. 深度策略(砍1-2方向，剩2个打⭐⭐⭐)
**决策**：**B的变体——2个⭐⭐⭐深度方向 + 1个⭐轻量方向**

| 方向 | 档位 | 定位 |
|------|------|------|
| 多模态(4.2) | ⭐⭐⭐拍照找货 | 深度方向1 |
| 对话智能(4.3) | ⭐⭐反选排除 + ⭐⭐⭐对比 | 深度方向2 |
| 业务闭环(4.1) | ⭐入门对话式加购 | 轻量覆盖方向 |
| 工程质量(4.4) | 不做 | 不投入 |

**理由**：官方说"做精一项胜过浅尝三项"。工程质量加分(缓存/首屏优化)在3周赛制里性价比低，SSE本身已解决流式体验。

## Q9: 数据策略

> **⚠️ 已被 2026-05-20 数据品类 Pivot 决策覆盖**：最终决策为只用官方100条，不自构造玩具数据。详见 [数据品类Pivot](2026-05-20-数据品类Pivot.md)。

**背景**：官方要求50-100条脱敏电商数据，字段含商品名、类目、价格、详情描述、主图URL。用户原有100-200条且定义了丰富玩具专属字段。
**选项**：50 / 100 / 150-200
**决策**：**80条 + 双数据源**
- 导师提供的脱敏电商数据（5基础字段，兼容通用）
- 自主构造80条（含玩具专属字段：age_min/age_max、safety_features、education_dimensions等）
**理由**：80条在50-100范围内偏多，足够覆盖3-4个玩具子类目。不管导师数据有没有扩展字段，自己的80条必须有——没有这些字段，硬过滤和反选排除就是假的。双数据源还展示了"兼容通用电商数据"的能力。

## Q10: P2 和砍掉清单

**背景**：P2是最后的打磨。有些东西需要彻底砍掉。
**决策**：彻底砍掉：语音输入、Chunk增删改查、Prompt优化Diff视图、知识图谱增强、评论情感摘要、热门查询缓存、首屏极速优化、端侧体验打磨、完整Web后台、支付/下单确认。P2只保留：Thinking骨架屏、卡片交互打磨、评测页面最小版、4条Demo稳定性打磨。
**原则**：先砍，有余力再考虑恢复。

## Q11: 交付材料对齐

**背景**：官方要求3项交付：可运行Demo、源代码仓库、技术文档。代码规范要求前后端分目录、依赖管理固定版本、核心逻辑注释。
**选项**：按官方建议重命名目录(backend→server等) / 保持现有结构只补要求
**决策**：**保持现有目录结构，只补2件事**：依赖版本锁定(requirements.txt固定版本) + 核心逻辑注释(RAG链路/Prompt构造)
**理由**：官方"如/client、/server、/docs"只是建议不是硬性要求。保持现有结构避免无意义的文件移动。

## Q12: 减分项防御

**背景**：官方4个减分项，用户只识别了2/4。
**决策**：补全防御措施：
| 减分项 | 防御 |
|--------|------|
| AI幻觉 | 混合检索+硬过滤+证据绑定(已有) |
| 纯Web/H5 | Android原生(已有) |
| Demo无法运行 | docker-compose up一键启动 + README运行说明(新增) |
| 无法解释代码原理 | design-decisions.md + 答辩前自读(新增) |

新增 `design-decisions.md`：每个核心决策3句话——选了什么、为什么、反过来会怎样。

## Q13: 双轨模型切换机制

**背景**：双轨策略需要2个不同API base URL、2套API Key、按任务类型选择endpoint、fallback机制。
**选项**：A. 配置层切换(dict+call_llm()) / B. 策略模式+抽象层(LLMRouter)
**决策**：**A. 配置层切换**
**理由**：3周比赛，配置层切换是最笨但最清晰的方案。一个dict配置+一个简单call_llm(task_type, prompt)函数，primary失败重试fallback，完事。不需要抽象层、策略模式、工厂。超过3层缩进就该重新设计。

## Q14: Demo路径对齐官方场景

> **已被 2026-05-20 数据品类 Pivot 决策覆盖**：当前 Demo 路径改为美妆护肤为主："推荐适合油皮的洗面奶，200元以内"、上传护肤品图片+"这个适合敏感肌吗？"、"不要含酒精的防晒霜"+"预算降到200"、"把这个加到购物车"。详见 [数据品类Pivot](2026-05-20-数据品类Pivot.md)。

**背景**：官方定义8个典型场景(基础/进阶/高级)，用户原有3条Demo路径。场景1太重(同时演示条件筛选+澄清+反选+预算)。
**决策**：**4条Demo路径，每条只演示1-2个核心能力**

| # | 路径 | 核心能力 | 对应官方场景 |
|---|------|---------|-------------|
| 1 | "推荐适合4岁孩子的益智玩具" | 模糊推荐+条件筛选 | 单轮模糊推荐+条件筛选 |

| 4 | "把这个加到购物车" | 对话式加购 | 购物车⭐入门 |

4条路径3.5分钟，完美匹配官方"3-5分钟Demo+技术讲解"。场景化组合推荐砍掉——80条数据撑不住跨类目组合。

## Q15: SSE事件协议——购物车新增事件

**背景**：加购是中断型操作(用户看到product_card后发起)，不是Agent主链路的一部分。需要决定是新增独立事件还是复用text_delta。
**选项**：方式1.新增cart_action事件 / 方式2.复用text_delta+前端正则检测
**决策**：**方式1.新增cart_action事件**
**理由**：加购是结构化操作不是文本对话，它改变客户端状态(购物车图标数字、商品卡"已加购"标记)。用独立事件前端不需要从文本做正则匹配。符合Agent-to-UI设计哲学——每种能力有独立事件类型和卡片渲染。

新增意图类型：add_to_cart、view_cart。
新增SSE事件：cart_action(action: add/remove/view, product_id, cart_id, quantity, status: success/failed)。

## Q16: 时间线验收标准

**背景**：P0/P1/P2逻辑改为评审权重驱动后，验收标准也要变。
**决策**：

| 时间点 | 验收标准 | 对应评审维度 |
|--------|---------|-------------|
| 第7天 | 评委拿起App能走通完整链路 + 一键启动 | 基础完整性35% + 工程底线25% |
| 第14天 | 4条Demo路径全部可演示 + 无幻觉无Bug | 效果可靠性20% + 加分项20% |
| 第18天 | 冻结功能 + design-decisions.md完成 | 全维度 |

铁律不变：第7天没跑通，P1/P2全砍只追P0。
