# BuyPilot-AI 决策记录（2026-05-20）：数据品类 Pivot

> 背景：探索导师提供的官方数据后发现，官方数据覆盖美妆护肤/数码电子/服饰运动/食品生活 4 个品类，完全不包含玩具。原选的"亲子玩具"唯一品类与官方数据零交集。同时官方数据的品类与课题说明会列举的典型用户场景完全一致。且官方数据质量（中文、含 rag_knowledge 3 类知识、本地图片）远超自构造数据。最终决策：彻底放弃 Amazon 数据和玩具自构造，只用官方 100 条。

---

## Q17: 数据源——官方数据 vs Amazon 玙具 vs 混合

**背景**：
- 官方数据：100 条，4 品类×25，中文，含 marketing_description + FAQ + reviews + SKUs + 本地图片，RAG 知识结构极丰富
- Amazon 数据：6484 条 Toys & Games，英文，仅 product_text blob，无玩具专属字段，需翻译+下载图片+补充字段
- 课题说明会官方场景："油皮洗面奶"(美妆) / "蓝牙耳机"(数码) / "推荐跑鞋"(服饰) — 与官方数据品类完全匹配

**选项**：A. 坚持 亲子玩具 + Amazon / B. 官方数据为主 + 自构玩具补充 / C. **只用官方 100 条**

**决策**：**C**

**理由**：
1. 官方数据开箱即用：中文+本地图+rag_knowledge 天然 chunking。任何自构造数据的工作量都是纯消耗，不增加评审得分。
2. 课题场景与官方品类完美匹配——评委会拿"油皮洗面奶""蓝牙耳机"测试，我们的 App 正好有这些品类的数据。
3. Amazon 数据需要翻译+字段补充+图片下载+chunk 拆分，流程复杂且质量不可控。3 周赛制下这些工作量是纯浪费。
4. 反选排除在美妆品类更真实——"不要含酒精的"需要语义理解，不是 SQL 字段过滤，更能体现 RAG+LLM 协同能力。

## Q18: 产品定位 Pivot

**背景**：原定位"把用户模糊购物需求转化为可解释决策路径的亲子玩具专家级决策智能体"，现在品类从唯一玩具变为多品类通用品类。

**决策**：从"亲子玩具专家"改为"多品类智能导购"。

**新定位**：把用户模糊购物需求转化为可解释决策路径的多品类智能导购决策智能体

**理由**：课题原标题就是"基于 RAG 的多模态电商智能导购 AI Agent"。多品类定位是回归课题原意，不是退让。

## Q19: 数据总量

**背景**：课题要求 50-100 条，官方数据正好 100 条。

**决策**：**100 条，仅使用官方数据**

**理由**：100 条在课题 50-100 要求范围内。4 品类×25，每个品类覆盖多个子类目，足够支撑 Demo 和评测。不再自构造任何补充数据——降低复杂度。

## Q21: 数据入库 Schema 适配

**背景**：官方数据 JSON 结构与后端 PRD 的 products 表 Schema 需要映射。

**官方数据字段 → products 表映射**：
- product_id → id
- title → name
- brand → brand
- category → category
- sub_category → sub_category
- base_price → price
- image_path → image_urls（指向本地图片路径）
- skus → 扁平化到 metadata JSONB
- rag_knowledge → 拆分为 product_chunks 表的多条 chunk（marketing_description / FAQ / reviews 各为独立 chunk）

**决策**：**products 表不增加玩具专属字段，用 metadata JSONB 承载品类属性**

**理由**：没有玩具数据了，age_min/age_max/safety_features 这些 nullable 字段全是 NULL——4 个 NULL 列比一个 JSONB 列更糟。metadata JSONB 承载品类属性（如护肤品的肤质适用、数码的存储规格、服饰的运动类型），一张表、没有特殊情况。
