## 2026-05-19 初轮决策

### 1. 砍掉 Stream SDK

**日期**：2026-05-19
**背景**：Android PRD 原计划使用 Stream Chat Android Compose SDK 作为聊天容器，同时用 OkHttp SSE 连接后端，形成两套消息流需要合并。
**选项**：保留 Stream SDK / 纯 OkHttp SSE + 自研 LazyColumn + ChatUiNode
**决策**：纯 OkHttp SSE + 自研 LazyColumn + ChatUiNode
**理由**：Stream SDK 引入双通道异步消息流，需要额外合并逻辑，增加复杂度且对三周 MVP 无额外价值。自研方案链路单一（OkHttp SSE → LazyColumn 渲染），数据结构清晰（sealed interface ChatUiNode），出错好定位。

---

### 2. 商品数据源：Amazon 多模态 RAG 数据集

**日期**：2026-05-19
**背景**：需要 200-500 条亲子玩具商品数据用于 RAG 检索验证。
**选项**：手动录入 / 爬虫抓取 / AI 生成种子数据 / 使用开源数据集
**决策**：使用 [Amazon-Multimodal-RAG-Assistant](https://github.com/easonwangzk/Amazon-Multimodal-RAG-Assistant) 的 `amazon_multimodal_clean.csv`（9509 条商品，其中 6488 条 Toys & Games）
**理由**：数据质量远高于手工录入，价格已为数字格式，有图片 URL 和多模态描述文本，开箱即用。

---

### 3. 图片全量下载存本地

**日期**：2026-05-19
**背景**：6488 条玩具商品有 Amazon CDN 图片 URL，但比赛演示时外链可能失效。
**选项**：全量下载 / 按需下载 500 条
**决策**：全量下载，存储于 `data/images/{uniq_id}.jpg`
**理由**：比赛演示需要 100% 可控，不依赖 Amazon CDN。一次性下载虽然占空间，但避免了演示时图片 403 的灾难。

---

### 4. 后端分模块架构

**日期**：2026-05-19
**背景**：FastAPI 后端需要 3 人并行开发，需要确定文件组织方式。
**选项**：单文件 main.py / 按职责分模块 / 先单文件后拆分
**决策**：从一开始就分模块（`routers/`、`services/`、`models/`、`db/`）
**理由**：3 人需要第一天就并行开发，分模块避免互相冲突。3 周赛制虽然时间紧，但模块化不会比单文件慢太多。

---

### 5. 前端 1 人独立负责

**日期**：2026-05-19
**背景**：3 人团队中前端需要几个人。
**选项**：前端多人分工 / 前端 1 人独立负责
**决策**：前端 1 人负责，后端提供接口即可
**理由**：Android 端主要是 UI 渲染 + SSE 连接，工作量集中在 Compose 组件和状态管理，1 人足够。后端 2 人可以更好地分工 LLM 调用和检索调优。

---

### 6. 后端 2 人分工：主开发 + 算法

**日期**：2026-05-19
**背景**：后端 2 人如何切分 FastAPI + RAG + LLM 调用的工作。
**选项**：按管道阶段切 / 按职责切（主开发 + 算法）
**决策**：人 A（主开发）负责 FastAPI 路由 + SSE 管道编排 + 接口契约 + 全链路串联；人 B（算法/检索）负责 LLM 调用封装 + pgvector 搭建 + Rerank 调优 + Prompt 优化
**理由**：SSE 编排需要唯一 owner，数据流清晰。3 周赛制需要一个对整体数据流负责的人。

---

### 7. SSE 管道：async generator 模式

**日期**：2026-05-19
**背景**：后端 `/chat/stream` 需要串联 5-6 个处理步骤（意图识别 → 购买标准 → 检索 → 生成 → 返回），如何组织。
**选项**：线性 async/await 链 / asyncio.gather 并行 / async generator 管道
**决策**：async generator 管道，每个 stage 为独立的 `async def yield_events(input)` 生成器
**理由**：每个 stage 独立可插拔、可单测。代码结构清晰：编排逻辑就是 `async for event in stage_n(stage_n-1(...))`，符合 Linus"消除特殊情况"原则。

---

### 8. 开发环境：Docker Compose

**日期**：2026-05-19
**背景**：3 个人本地开发，环境一致性怎么保证。
**选项**：Docker Compose / 本地装 PG / SQLite 开发
**决策**：Docker Compose 统一环境（PostgreSQL + pgvector + FastAPI）
**理由**：pgvector 是核心依赖，不能用 SQLite 替代。3 人同一套 PG 版本，避免"我本地能跑"问题。

---

### 9. 商品筛选：LLM 批量打分

**日期**：2026-05-19
**背景**：6488 条 Amazon 商品数据需要筛出 ~500 条适合亲子玩具导购的高相关商品。
**选项**：关键词硬过滤 / LLM 批量打分
**决策**：用 Qwen-Turbo 对 6488 条商品逐条打 0-1 分（是否适合亲子玩具导购），取 Top 500 + 人工复核
**理由**：语义理解准确率远高于关键词规则，Qwen-Turbo 便宜快速（<10 元，20-30 分钟跑完）。打分过程本身可作为 RAG pipeline demo 给评委看。

---

### 10. 数据库：8 张表一次性建好

**日期**：2026-05-19
**背景**：PRD 定义了 8 张表（products、product_chunks、embeddings、sessions、messages、feedbacks、retrieval_traces、eval_results），是分批建还是一次性建。
**选项**：Day 1 建核心 3 张 / 一开始全建
**决策**：8 张表一次性建好
**理由**：避免后期 migration 踩坑。SQLModel 一次性定义所有 model 后 `CREATE TABLE` 很快，后续只加字段不改表结构。

---

### 11. ChatUiNode 数据结构：sealed interface

**日期**：2026-05-19
**背景**：前端 ChatUiNode 是所有渲染元素的基类，数据类型怎么定义。
**选项**：sealed interface + data class / 统一 base class + type 字段
**决策**：sealed interface + data class 枚举
**理由**：Kotlin when 表达式编译期穷尽检查，少一个分支编译不过，比运行时崩溃好一万倍。前端只有 1 人，不存在多人改冲突。

---

### 12. 管道降级策略

**日期**：2026-05-19
**背景**：async generator 管道中任意 LLM 调用失败或检索为空时，SSE 流已开始，不能直接断开。
**选项**：详见下方决策
**决策**：
- 意图识别失败 → 跳过意图，直接用原始 query 做检索
- 检索结果为空 → 放宽条件重试（如去掉安全约束再搜）→ 仍为空 → 坦诚告知用户"暂时没找到匹配商品"，引导换问法
**理由**：cascading fallback 策略比直接报错或返回硬编码假推荐好。评委能看出坦诚的降级比假的推荐更有诚意。

---

### 13. 图片上传：两步走

**日期**：2026-05-19
**背景**：用户拍玩具照片问"这个适合 3 岁孩子吗？"，图片怎么传给后端。
**选项**：先上传再聊天（两步）/ 图片 base64 随聊天一起发（一步）
**决策**：先 POST `/upload/image` 拿到 `image_url`，再把 URL 放入 `/chat/stream` 请求体
**理由**：图片上传和 AI 处理解耦，上传可独立做进度条、压缩、重试。SSE 连接不需要等图片传完才建立。职责清晰。

---

### 14. LLM 调用：Profile 配置驱动

**日期**：2026-05-19
**背景**：后端要调百炼 4-5 个模型（Qwen-Turbo、Qwen-Plus、Qwen-VL-Plus、text-embedding-v3、gte-rerank），怎么组织调用。
**选项**：统一 client / 按模型独立封装 / 统一 client + Profile
**决策**：一个 `BailianClient` + YAML 配置定义多个 profile（intent、generation、vision），调用时 `client.chat(prompt, profile="intent")`
**理由**：换模型调 prompt 时只改配置不动代码。每个 profile 有独立的 model 名、temperature、max_tokens、timeout。配置驱动比代码驱动靠谱。

---

### 15. 演示环境：本地笔记本 Docker Compose

**日期**：2026-05-19
**背景**：有内网高性能服务器，但无公网 IP 和域名。比赛现场演示时后端跑在哪。
**选项**：本地笔记本 / 云服务器 / 内网穿透
**决策**：本地笔记本跑 Docker Compose，Android 走同一 WiFi（或手机热点）
**理由**：LLM 调用在百炼云端，本地只跑 PG + FastAPI，2C4G 完全够用。零外部网络依赖，出问题直接看日志。内网穿透多一层不确定性，比赛不应该依赖。
